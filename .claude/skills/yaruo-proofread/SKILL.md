# やる夫式教材 校正・検証スキル

対象は yaruo-rediscovery が生成した会話形式の教材（6〜10万字）。処理は2つ。順番は必ず **1→2**。

**前提**：書式整形（台詞インデント・強調記号 `**` の修正）は yaruo-format スキルの担当で、ユーザーが個別に起動する。このスキルからは実行しない（整形の崩れに気づいても報告に載せるだけでよい）。

ファイルが大きいので一度に全部読もうとしない。最初に見出し（`grep -n '^## '`）で幕構成を把握し、以降は幕単位で読み書きする。作業メモ（主張リスト・脚注の通し番号・修正記録）はスクラッチパッドに置き、幕をまたいで維持する。

## 処理1: ファクトチェックと参考文献

### 対象の選別

チェック対象は **事実・理論・歴史について検証可能な主張をしている発言**：固有名詞（人物・研究・機関）、年号、統計値・実験値、定理や発見の帰属、歴史的経緯。キャラクターの感想・比喩・純粋な論理展開・数式変形そのもの（処理2の担当）は対象外。

**最重要の注意**：この教材形式は「生徒がわざと間違える」構造（素朴案→反例→修正の再発見サイクル）を持つ。**やる夫の素朴案や誤答は物語装置であり、訂正してはならない。** 直後にやらない夫が反例で潰していれば、それは意図された誤りである。訂正対象は「教材が正しいと提示している記述」──主にやらない夫の解説と、最終的に到達した結論──に限る。

### 手順

1. **全幕を通して読み**、検証可能な主張を作業メモに一括で列挙する（発言者・行番号・主張の要約）。Web 検証は幕ごとに分けず、主張を出し切ってからまとめて処理する。
2. 列挙した主張を **想定ソース（citする予定のページ）ごとにグルーピング** する。1ページが複数主張をカバーすることが多い（例：「Klein–Gordon equation」1ページでシュレーディンガーの相対論版と Pauli–Weisskopf の再解釈の両方が確認できる）。同一ソースは同じ脚注番号を再利用するので、この時点で脚注番号を割り当てておく。
3. **Web 検証はインラインで行うが、ページ本文をこの校正のコンテキストに持ち込まない。** ソースグループごとに次の2段構えで検証し、結果（判定・訂正/改善案・URL・ページタイトル・サイト名）を作業メモに主張と対応づけて保存する（判定＝処置の分岐、訂正・改善案＝修正の下敷き、URL/タイトル/サイト名＝脚注定義）。
   - **第1段：Wikipedia の summary API（トークンほぼゼロ）。** 想定ソースが Wikipedia（定番の事実・発見・年号・帰属はほぼこれ）なら、まず `curl -s "https://en.wikipedia.org/api/rest_v1/page/summary/<記事名>" | jq '{title, extract, url: .content_urls.desktop.page}'`（日本語版は `ja.wikipedia.org`）で実在・正式タイトル・要旨を取る。**要旨（extract）だけで裏が取れる主張はここで完了**——脚注に必要な URL・タイトルも同時に揃う。
   - **第2段：WebFetch（要旨で足りない主張・Wikipedia 以外のソース）。** WebFetch は本文を返さず、**小型高速モデルが `prompt` に答えた結果だけ**を返す。だから 1ソース1回、prompt に主張リストと出力形式を明記して呼ぶ：「以下の各主張について (a) 判定（本文が裏付ける/本文と矛盾する/本文に記載なし）、(b) 判定に関わらず主張をより正確・明確にする訂正・改善案（不要なら『改善案なし』）、(c) ページの正式タイトル、を簡潔に返せ」。長文引用を求めない。
   - **並列実行で処理時間を短縮する。** 互いに独立した curl / WebFetch は 1メッセージにまとめて同時に発行する。
   - citするページが分からない・主張が本当に怪しいときだけ WebSearch を使う（検索結果はかさばるので最後の手段）。フェッチ先は確実に開けるドメインを選ぶ：`en.wikipedia.org` はほぼ確実に開けるが、`home.cern` `nobelprize.org` 等は 403/リダイレクトで空振りしやすい。公式サイトを citしたくても、まず Wikipedia で事実確認し Wikipedia を出典にするのが堅実。
   - **記載する URL は必ず curl か WebFetch で実在確認したものに限る。開いていない URL を書いてはならない（捏造は最悪の失敗）。**
4. 「記載なし」だった主張のうち、教材の結論に関わる重要なものだけ追加調査する（それ以外は保留）。
5. 判定に応じて処置する。**どの判定でも「訂正・改善案」に目を通し、教材を良くするものは会話の文体（語尾・キャラクター）と再発見サイクルの構造を壊さない範囲で本文に反映する**（案は素材であって、そのまま貼るのではなく台詞になじませる。反映するかは自分で判断し、採否と内容を報告に記録する）：
   - **裏付けあり** → 該当する文の直後（台詞内、閉じ括弧の前）に脚注マーカー `[^N]` を付ける。改善案が有用なら表現を磨く。
   - **矛盾（誤り）** → 訂正案を下敷きに最小限の修正を行い、修正後の記述に脚注を付ける。修正内容は報告に記録する。
   - **確認できない** → ソースを捏造しない。「〜と言われている」への弱め方や削除を検討し、判断が難しければ報告で保留として挙げる。

   **注意（再発見サイクル）**：やる夫の素朴案・誤答も検証すれば「誤り」と出るが、それらは物語装置なので訂正を反映してはならない（対象の選別を参照）。訂正・改善案を反映するのは「教材が正しいと提示している記述」に限る。

### 参考文献の書式

脚注番号はファイル全体の通し番号。定義はファイル末尾に `---` 区切りで `## 参考文献` 節を設けてまとめる。書式は厳守：

```markdown
[^2]: [サイト名『ページタイトル』](URL), 2026年7月8日参照.
```

参照日は **作業当日**（`date "+%Y年%-m月%-d日"` で取得）。サイト名は運営主体（例：国立天文台、NASA、日本銀行）、ページタイトルは実際のページの題名。同じソースを複数箇所で使う場合は同じ脚注番号を再利用する。

## 処理2: 数式と会話の整合性検証

幕ごとに全数式（`$$...$$` と台詞中の `$...$`）を抽出し、次の4点を確認する。

1. **式単体の正しさ**：係数・符号・添字・次元（単位）が正しいか。標準的な教科書の形と一致するか。
2. **式変形の連鎖**：前の式から次の式が本当に導けるか。省略されたステップを自分で埋めて確認する。手計算で追えないものは Python（sympy / numpy）を書いて検算する。
3. **会話と式の一致**：台詞が述べている内容と直後の式が一致しているか。「この式の右辺は〜」のような言及が実際の式と食い違っていないか。数値例は必ず実際に計算して合っているか確かめる（このスキルの検出力が最も高いのはここ。「62,800 km のはずが〜」型の具体的数値は全部再計算する）。
4. **記号の一貫性**：同じ物理量にファイル全体で同じ記号が使われているか。途中で定義なく記号が変わっていないか。

ここでも再発見サイクルに注意：**生徒の誤った式は、直後に反例で棄却されているなら正しい誤りである。** 修正対象は (a) 正解として提示された式の誤り、(b) 「わざと間違えたはずの式」が実は間違っておらず反例が成立していない場合、(c) 会話が式の内容を誤って説明している場合。

検算スクリプトはスクラッチパッドに書き、削除せず残す（報告から参照できるように）。

## 完了報告

全処理後、ユーザーに以下を報告する：

- ファクトチェック：検証した主張の数／付与した脚注の数／**修正した記述の一覧（修正前→修正後）**／改善案を採用しなかった主張とその理由／確認できず保留にした主張
- 数式検証：検証した式の数／**修正した式の一覧（修正前→修正後、根拠）**／検算に使ったスクリプトの場所

修正が1件もない処理についても「検証した上で問題なし」と明示する（やっていないのと区別がつくように）。

## チェックリスト（納品前に自己検査）

- [ ] Web 検証でページ本文をコンテキストに持ち込んでいないか（summary API か WebFetch の prompt 回答だけで済ませたか）
- [ ] 記載した URL はすべて curl / WebFetch で実在確認済みのもの（自分で捏造・改変していない）か
- [ ] 参照日は作業当日か（書式：`YYYY年M月D日参照.`）
- [ ] 脚注番号は通し番号で、重複・欠番がないか
- [ ] 再発見サイクルの「意図された誤り」を訂正してしまっていないか
- [ ] 修正した台詞はキャラクターの語尾・文体を維持しているか
- [ ] 具体的数値の登場箇所をすべて再計算したか
- [ ] 修正一覧を修正前→修正後の形で報告したか
