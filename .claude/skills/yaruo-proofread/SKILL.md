# やる夫式教材 校正・検証スキル

対象は yaruo-rediscovery が生成した会話形式の教材（6〜10万字）。処理は2つ。順番は必ず **1→2**。

**前提**：書式整形（台詞インデント・強調記号 `**` の修正）は yaruo-format スキルの担当。校正を始める前に yaruo-format を先に実行しておくこと（整形で行が動くと、校正中の編集の差分が読みにくくなるため）。未実施ならまず yaruo-format を実行する。

ファイルが大きいので一度に全部読もうとしない。最初に見出し（`grep -n '^## '`）で幕構成を把握し、以降は幕単位で読み書きする。作業メモ（主張リスト・脚注の通し番号・修正記録）はスクラッチパッドに置き、幕をまたいで維持する。

## 処理1: ファクトチェックと参考文献

### 対象の選別

チェック対象は **事実・理論・歴史について検証可能な主張をしている発言**：固有名詞（人物・研究・機関）、年号、統計値・実験値、定理や発見の帰属、歴史的経緯。キャラクターの感想・比喩・純粋な論理展開・数式変形そのもの（処理2の担当）は対象外。

**最重要の注意**：この教材形式は「生徒がわざと間違える」構造（素朴案→反例→修正の再発見サイクル）を持つ。**やる夫の素朴案や誤答は物語装置であり、訂正してはならない。** 直後にやらない夫が反例で潰していれば、それは意図された誤りである。訂正対象は「教材が正しいと提示している記述」──主にやらない夫の解説と、最終的に到達した結論──に限る。

### 手順（幕ごと）

1. 幕を読み、検証可能な主張を作業メモに列挙する（発言者・行番号・主張の要約）。
2. 各主張を WebSearch / WebFetch で検証する。ソースの優先順位：一次資料（原論文・公的統計・機関の公式ページ）＞ 査読論文・学術機関の解説 ＞ 定評ある百科事典・大手報道。まとめサイト・個人ブログ・AI生成ページは使わない。
3. 記載する URL は必ず WebFetch で開き、ページが実在しタイトルが合っていることを確認する。**確認していない URL を書いてはならない（捏造は最悪の失敗）。**
4. 判定に応じて処置：
   - **正しい** → 該当する文の直後（台詞内、閉じ括弧の前）に脚注マーカー `[^N]` を付ける。
   - **誤り** → 会話の文体（語尾・キャラクター）と再発見サイクルの構造を壊さない最小限の修正を行い、修正後の記述に脚注を付ける。修正内容は報告に記録する。
   - **確認できない** → ソースを捏造しない。記述を「〜と言われている」程度に弱めるか削除を検討し、判断が難しければ報告で保留として挙げる。

### 参考文献の書式

脚注番号はファイル全体の通し番号。定義はファイル末尾に `---` 区切りで `## 参考文献` 節を設けてまとめる。書式は厳守：

```markdown
[^2]: [サイト名『ページタイトル』](URL), 2026年7月8日参照.
```

参照日は **作業当日**（`date "+%Y年%-m月%-d日"` で取得）。サイト名は運営主体（例：国立天文台、NASA、日本銀行）、ページタイトルは実際のページの題名。同じソースを複数箇所で使う場合は同じ脚注番号を再利用する。

## 処理2: 数式と会話の整合性検証

幕ごとに全数式（`$$...$$` と台詞中の `$...$`）を抽出し、次の4点を確認する。

1. **式単体の正しさ**：係数・符号・添字・次元（単位）が正しいか。標準的な教科書の形と一致するか。
2. **式変形の連鎖**：前の式から次の式が本当に導けるか。省略されたステップを自分で埋めて確認する。手計算で追えないものは Python（sympy / numpy）を書いて検算する。
3. **会話と式の一致**：台詞が述べている内容と直後の式が一致しているか。「この式の右辺は〜」のような言及が実際の式と食い違っていないか。数値例は必ず実際に計算して合っているか確かめる（このスキルの検出力が最も高いのはここ。「62,800 km のはずが〜」型の具体的数値は全部再計算する）。
4. **記号の一貫性**：同じ物理量にファイル全体で同じ記号が使われているか。途中で定義なく記号が変わっていないか。

ここでも再発見サイクルに注意：**生徒の誤った式は、直後に反例で棄却されているなら正しい誤りである。** 修正対象は (a) 正解として提示された式の誤り、(b) 「わざと間違えたはずの式」が実は間違っておらず反例が成立していない場合、(c) 会話が式の内容を誤って説明している場合。

検算スクリプトはスクラッチパッドに書き、削除せず残す（報告から参照できるように）。

## 完了報告

全処理後、ユーザーに以下を報告する：

- ファクトチェック：検証した主張の数／付与した脚注の数／**修正した記述の一覧（修正前→修正後）**／確認できず保留にした主張
- 数式検証：検証した式の数／**修正した式の一覧（修正前→修正後、根拠）**／検算に使ったスクリプトの場所

修正が1件もない処理についても「検証した上で問題なし」と明示する（やっていないのと区別がつくように）。

- [ ] 参照日は作業当日か（書式：`YYYY年M月D日参照.`）
- [ ] 脚注番号は通し番号で、重複・欠番がないか

- [ ] 具体的数値の登場箇所をすべて再計算したか
