# やる夫で学ぶデータベース設計 ── 給与計算システムを再発明する

---

## 登場人物

**やる夫** ── SQLと正規化を学んだばかりのエンジニア。知人の会社の給与管理システムを任された。設計とは正規化のことだと思っている。

**やらない夫** ── データベース設計の実務を長く歩いてきたエンジニア。答えを教えず、反例のデータと問いで導く。

## 第1幕　正規化が効かない日

---

**やる夫**：
「やる夫、SQLと正規化をマスターしたお。
　第1正規形から第3正規形、ボイス・コッド正規形まで説明できるお。
　知り合いのカモメ製作所（従業員30人）の給与管理を、
　Excelからデータベースに引っ越す仕事を請けたお」

**やらない夫**：
「ちょうどいい。そのExcelを見せてみろ」

**やる夫**：
「こんな感じだお」

| 社員番号 | 氏名 | 部署 | 基本給 | 手当1名称 | 手当1金額 | 手当2名称 | 手当2金額 | 振込口座 |
|--|--|--|--|--|--|--|--|--|
| 101 | 山田 | 製造部 | 250000 | 通勤 | 12000 | 住宅 | 20000 | ○○銀行… |
| 102 | 佐藤 | 経理部 | 310000 | 通勤 | 8000 | | | △△銀行… |

**やらない夫**：
「これをどうテーブルにする」

**やる夫**：
「ふっふっふ、待ってましたお。
　まず手当1、手当2の繰り返し列。これは第1正規形違反の典型だお。
　手当が3種類ある社員が来た瞬間に列が足りなくなるし、
　『通勤手当の合計』を出すのに手当1と手当2の両方を調べる羽目になるお。
　だから行に立てるお」

```sql
CREATE TABLE employees (
    emp_id     integer PRIMARY KEY,
    name       text NOT NULL,
    dept_name  text NOT NULL,
    base_salary numeric NOT NULL,
    bank_account text NOT NULL
);

CREATE TABLE employee_allowances (
    emp_id    integer REFERENCES employees,
    allowance_name text,
    amount    numeric NOT NULL,
    PRIMARY KEY (emp_id, allowance_name)
);
```

**やらない夫**：
「上出来だ。繰り返し項目を行に立てるのは、
　コッドが1970年の論文でリレーショナルモデルを提案したとき、
　最初に要求したことでもある[^1]。
　当時のCOBOLのレコードにはOCCURS句という『繰り返し項目』が普通にあって、
　その排除こそが第1正規形の出発点だった。

　一つだけ細かい指摘をする。基本給の型、`numeric` にしたな。
　`real` や `double precision`──浮動小数点にしなかった理由は言えるか」

**やる夫**：
「え。数値なんだからどっちでもいいんじゃないかお。
　浮動小数点のほうが速そうだお」

**やらない夫**：
「では計算してもらおう。浮動小数点で 0.1 を10回足せ」

**やる夫**：
「0.1 × 10 で 1.0 に決まってるお。
　……一応、実行してみるお」

```sql
SELECT 0.1::float8 + 0.1::float8 + 0.1::float8 + 0.1::float8 + 0.1::float8
     + 0.1::float8 + 0.1::float8 + 0.1::float8 + 0.1::float8 + 0.1::float8;
-- 結果: 0.9999999999999999
```

「**1.0 にならないお！？**」

**やらない夫**：
「浮動小数点は2進数の近似だ。0.1 は2進数では無限小数になるから、
　最初から正確には格納されていない[^2]。
　科学計算なら誤差の海の一滴だが、給与は1円ずれたら事故だ。
　金額は `numeric`（固定小数点）か、円の整数と決めて整数型。これは反射でやれ。

　ついでに聞く。月の途中で入社した社員の日割り給与、
　月給285,000円で所定21日のうち11日勤務なら？」

**やる夫**：
「285,000 × 11 ÷ 21 で……149,285.714…円。
　割り切れないお。切り捨て？　四捨五入？」

**やらない夫**：
「それはデータベースが決めることではない。**就業規則が決めることだ**。
　端数処理の規程を確認せずに ROUND を書いたら、それは設計者の越権だ。
　『技術で決めていいこと』と『業務が決めること』の境界線──
　この線は今後何度も出てくるから、最初の教訓として覚えておけ。

　さて、テーブルはできた。正規形も申し分ない。
　……ところが今朝、カモメ製作所から3件の連絡が来ている」

**やる夫**：
「なんだお。もう完璧なテーブルがあるお」

**やらない夫**：
「読み上げるぞ。

　**連絡1**：『製造部の山田ですが、今月から経理部の仕事も2割ほど
　兼務してもらうことになりました。労務費の配賦があるので比率も管理したいです』

　**連絡2**：『6月の給与計算前にすみません、山田の基本給、
　実は4月にさかのぼって250,000円から265,000円に昇給が決まりました。
　4月分と5月分の差額も6月に払います』

　**連絡3**：『3年前に退職した田中が、来月また入社することになりました。
　昔の社員番号って使い回していいんでしたっけ？』」

**やる夫**：
「……えっと。
　連絡1は、`employees.dept_name` に部署が1個しか入らないお。
　連絡2は、`base_salary` を265,000にUPDATEしたら
　『4月から』という情報がどこにも残らないお。
　連絡3は、そもそも退職した田中がテーブルにいるのかどうかも怪しいお。

　……どれも、正規形の話じゃ**ない**お？？
　第3正規形もBCNFも、この3件には何も言ってくれないお」

**やらない夫**：
「そこに気づけたなら今日の授業は始められる。
　正規化とは何をしてくれる道具だった？」

**やる夫**：
「決めた属性と関数従属性に対して、更新時異常が起きない置き方を教えてくれる……

　あ。**『決めた属性に対して』** だお。
　部署を1個持つか複数持つか、給与の歴史を持つか現在だけ持つか、
　田中を『同じ社員』と見るか『別の社員』と見るか──
　**何を事実として記録するかを決めるのは正規化の前の仕事**で、
　正規化はその後の整理整頓しかしてくれないんだお」

**やらない夫**：
「その通り。**正規化は文法だ。どんな文を書くべきかは教えてくれない。**
　そして今の3件の連絡は、偶然ではなく、
　データベース設計の三大難所に1件ずつ対応している。

　**問い I**：関係はいくつまで許すのか。──山田は何個の部署に属せるのか（**関係の自由度**）
　**問い II**：変化をどう記録するのか。──4月の給与はいくらだったのか（**時間**）
　**問い III**：何をもって『同じ』とするのか。──戻ってきた田中は誰なのか（**同一性**）

　この3つには、正規化のような機械的な答えはない。
　だが行き当たりばったりでもない。
　それぞれに **原則** があって、お前はこれからそれを1個ずつ自力で掘り出す」

**やる夫**：
「3つの原則……。
　その前に1個聞いていいかお。さっきから引っかかってるんだお。
　連絡2の昇給の件、UPDATEしたら歴史が消えるってことは、
　そもそもデータベースには何を入れるべきなんだお？
　**『今の状態』を入れるのかお。それとも『起きたこと』を入れるのかお**」

**やらない夫**：
「……それは今日一番の問いだ。
　だが答えは教えない。お前が第3幕と第7幕で、二段階に分けて自分で見つける。
　保留にしておけ。

　まずは連絡1、山田の兼務からだ」

## 第2幕　原則 I ── 関係の自由度

### 2-1　山田、二重生活を始める ── 多重度と関連の属性

---

**やる夫**：
「兼務なんて簡単だお。列を1個足せばいいお」

```sql
ALTER TABLE employees ADD COLUMN dept_name2 text;
```

「山田は `dept_name = '製造部'`、`dept_name2 = '経理部'` だお。
　シンプルイズベストだお」

**やらない夫**：
「その手をどこかで見なかったか。今朝のExcelを思い出せ」

**やる夫**：
「……あ。
　**手当1、手当2と同じ病気だお！！**
　3部署兼務する人が来たら `dept_name3` が要るし、
　『経理部に属する人の一覧』を出すのに
　`WHERE dept_name = '経理部' OR dept_name2 = '経理部'` って書く羽目になるお。
　自分で第1正規形違反をやらかすところだったお。行に立てるお」

```sql
CREATE TABLE departments (
    dept_id   integer PRIMARY KEY,
    dept_name text NOT NULL
);

CREATE TABLE dept_assignments (
    emp_id  integer REFERENCES employees,
    dept_id integer REFERENCES departments,
    PRIMARY KEY (emp_id, dept_id)
);
```

「従業員と部署の **多対多** を、間に1枚テーブルを挟んで表すお。
　ついでに部署もマスタテーブルに独立させたお。
　部署名の表記ゆれ（『製造部』と『製造 部』）で集計が割れるのは嫌だお」

**やらない夫**：
「いい判断だ。だがカモメ製作所の連絡には続きがあったな。
　『労務費の配賦があるので比率も管理したい』。
　山田の人件費、月265,000円のうち製造部の負担分はいくらだ」

**やる夫**：
「8割が製造だから 265,000 × 0.8 = 212,000円、経理が 53,000円……
　って、この **8割** はどこに格納するんだお？
　`employees` の列じゃないお。山田固有じゃなくて『山田と製造部の関係』の値だお。
　`departments` の列でもない。

　……そうか、置き場所は1個しかないお。**中間テーブルそのもの** だお」

```sql
CREATE TABLE dept_assignments (
    emp_id     integer REFERENCES employees,
    dept_id    integer REFERENCES departments,
    ratio      numeric NOT NULL,   -- 配賦比率
    is_primary boolean NOT NULL,   -- 主務か兼務か
    PRIMARY KEY (emp_id, dept_id)
);
```

**やらない夫**：
「**関連は、それ自体が属性を持ちうる**。
　1976年にチェンが実体関連モデル（ERモデル）を提案したときから、
　関連（relationship）は属性を持てる一人前の存在として扱われている[^3]。
　中間テーブルは多対多の苦肉の策ではなく、
　『所属』という事実そのものを1行1事実で記録する場所だ。
　発令日、辞令番号、比率──『所属についての事実』は全部ここに載る」

**やる夫**：
「1行1事実……。
　`(山田, 製造部, 0.8, 主務)` という行が『山田は製造部に主務として8割所属する』
　という **文** になってるんだお。
　第1幕の『どんな文を書くか』ってこういうことかお」

### 2-2　制約の射程 ── データベースはどこまで守ってくれるか

---

**やらない夫**：
「では、この設計を攻撃する。データを2行入れさせてもらう」

```sql
INSERT INTO dept_assignments VALUES (101, 1, 0.8, true);  -- 山田・製造部・主務
INSERT INTO dept_assignments VALUES (101, 2, 0.2, true);  -- 山田・経理部・主務
```

**やる夫**：
「……両方 `is_primary = true`。**主務が2つ** だお。
　人事制度上、主務はちょうど1つのはずだお。
　でも主キーは `(emp_id, dept_id)` だから、この2行はすんなり入っちゃうお」

**やらない夫**：
「どう防ぐ」

**やる夫**：
「`UNIQUE (emp_id, is_primary)` ……は駄目だお。
　`(101, false)` が2行あるだけで死ぬから、**兼務が2つ持てなくなる**お。
　縛りたいのは true のほうだけなんだお。
　……そんな都合のいい制約、あるのかお？」

**やらない夫**：
「PostgreSQLにはある。**部分一意インデックス**──条件つきの一意制約だ」

```sql
CREATE UNIQUE INDEX one_primary_per_emp
    ON dept_assignments (emp_id) WHERE is_primary;
```

**やる夫**：
「WHERE句つきのインデックス！
　`is_primary = true` の行だけの世界で `emp_id` が一意、ってことかお。
　検収するお。

　`(101, 製造部, true)` → 1件目、入る。✓
　`(101, 経理部, true)` → **一意制約違反で弾かれる**。✓
　`(101, 経理部, false)` → 兼務は入る。✓
　`(101, 総務部, false)` → 兼務2つ目も入る。✓

　**弾くべきものを弾き、通すべきものを通してる**お。合格だお」

**やらない夫**：
「では第二波だ。配賦比率のほうはどうだ。
　`(101, 製造部, 0.8)` と `(101, 経理部, 0.3)`──**合計1.1** の2行を入れるぞ」

**やる夫**：
「合計が1.0じゃないと配賦したとき人件費が水増しされるお！
　CHECK制約で…… `CHECK (ratio の合計 = 1.0)` ……

　……書けないお。
　CHECK制約は **1行の中身** しか見られないお。
　『山田の全行の合計』は行をまたいだ性質だから、CHECKの射程の外だお」

**やらない夫**：
「そうだ。ここで一度、武器庫の棚卸しをしておけ。
　データベースが宣言だけで守ってくれる整合性には、届く範囲がある。

| 守りたいこと | 使う道具 | 射程 |
|--|--|--|
| 値が変（負の給与） | CHECK制約 | 1行の中 |
| 重複（同じ所属が2行） | 主キー・UNIQUE | 1列集合の一致 |
| 条件つき重複（主務2つ） | 部分一意インデックス | 条件を満たす行の中 |
| 迷子の参照（存在しない部署） | 外部キー | 2テーブル間 |
| 行をまたぐ集計（比率合計=1.0） | **宣言的制約では不可** | トリガーかアプリで守る |

　最後の行が重要だ。『比率の合計は1.0』のような行またぎの不変条件は、
　トリガーで頑張るか、アプリケーションの更新処理を一本化して守るしかない。
　**どの不変条件をDBに預け、どれをアプリが背負うか**──これも設計の決断だ」

**やる夫**：
「制約って『あるだけ全部つける』もんだと思ってたお。
　届く範囲を測ってから配置する武器だったんだお」

### 2-3　「たまに複数」問題 ── 未来への賭け方

---

**やらない夫**：
「次はもっと嫌な問題だ。振込口座。
　現状、`employees.bank_account` で1人1口座。
　カモメ製作所の30人は全員1口座だ。……このままでいいか？」

**やる夫**：
「全員1個なんだから1列でいいお。
　……と言いたいけど、さっき部署で殴られたばかりだから警戒するお。
　でも部署と違って、今回は **現実に複数の人がいない** お。
　いない未来のためにテーブルを増やすのは過剰設計な気もするお……。

　どっちだお。やる夫には決め手がないお」

**やらない夫**：
「良い状態だ。**どちらの案にも本物の言い分がある**──それが設計の通常運転だ。
　決め手は理屈ではなく、コストの見積もりで作る。
　両方の未来を最後まで想像しろ。

　**未来A**：1列で作った。2年後、『財形貯蓄で2口座に分けたい』社員が現れた。
　何をする？　作業を全部列挙しろ」

**やる夫**：
「えーと。
　1. `bank_accounts` テーブルを新設する
　2. 既存30人ぶんの `bank_account` 列のデータを新テーブルに移送する
　3. 元の列を消す
　4. `employees.bank_account` を読んでる **全部のSQLとアプリコード** を探して書き換える
　5. 振込データ作成処理を『1人複数行』前提に直して、テストし直す

　……4と5が重いお。『全部探す』って、漏れたら本番で落ちるやつだお」

**やらない夫**：
「**未来B**：最初から1対多のテーブルで作った。
　だが2口座の社員は結局10年間現れなかった。毎日何を払い続ける？」

**やる夫**：
「口座を取るたびにJOINが1個増える。
　振込処理は『複数行あるかもしれない』前提で書くから微妙に複雑になる。
　新人が『なんで口座だけテーブル別なんですか』って聞いてくる。
　……地味な税金を払い続ける感じだお。

　並べてみると **痛みの種類が違う** お。
　未来Aの痛みは一発の移行手術で、失敗リスクつき。
　未来Bの痛みは薄く広い日常コスト。
　**しかも非対称だお**。多対応のテーブルを後から1個に縛るのは
　制約を1本足すだけだけど、1列を後から多対応にするのは今の手術一式だお」

**やらない夫**：
「その非対称性が原則の半分だ。もう半分は、賭けの確率を上げる技術──
　つまり **ドメインの人への聞き方** だ。
　カモメ製作所の総務さんに『口座が複数の人っていますか？』と聞いたら
　何と返ってくると思う」

**やる夫**：
「『いないですね』だお」

**やらない夫**：
「では『もし社員から2口座に分けたいと言われたら、今どうしてますか』と聞いたら？」

**やる夫**：
「……『あー、部長の鈴木さんだけ特別で、
　2つ目はExcelの備考欄にメモして手動で振り込んでます』
　みたいな答えが返ってきそうだお。

　！！　**『いない』んじゃなくて『システムの外に追い出されてる』** だけかお！」

**やらない夫**：
「現場のExcel・備考欄・付箋は、モデルから漏れた多重度の死体置き場だ。
　『ありえますか』はモデルの中の話しか答えてもらえない。
　『そのときどうしてますか』は運用の実態を吐かせる。
　例外は宣言の中ではなく、**運用の隅に埋まっている**。

　さて、口座はどうする」

**やる夫**：
「テーブルは1対多で作るお。ただし未来Bの日常コストも下げたいお。
　……そうだ、こうするお」

```sql
CREATE TABLE bank_accounts (
    emp_id   integer REFERENCES employees,
    priority integer NOT NULL,       -- 1が主口座
    account  text NOT NULL,
    amount_rule text NOT NULL,       -- '全額' or 固定額
    PRIMARY KEY (emp_id, priority)
);

-- 日常業務のための「1人1口座」の顔
CREATE VIEW primary_accounts AS
    SELECT emp_id, account FROM bank_accounts WHERE priority = 1;
```

「**記録は1対多で広く持って、ビューで1対1の狭い顔を見せる**お。
　ほとんどのコードは `primary_accounts` を読めばよくて、
　2口座対応が本当に必要になったら振込処理だけ本体を読むお」

**やらない夫**：
「賭け金の圧縮まで自力で辿り着いたか。完璧だ。
　では第2幕の成果を、お前の言葉で原則にまとめろ」

**やる夫**：
「**原則 I（関係の自由度）**：

　**多重度は『今の現実の観察』では決まらない。未来への賭けだお。**
　**そして賭けは、外れたときの痛みが小さい側に張る。**
　**狭く作って外す痛みは移行手術一式、広く作って外す痛みは日常の複雑さ。**
　**この2つは非対称で、たいてい移行のほうが高くつく。**
　**迷ったら記録は広く取り、制約とビューで狭く見せて、賭け金を減らすお。**

　あと、聞き込みの技術も原則の一部だお。
　『ありえますか』じゃなくて『そのときどうしてますか』だお」

**やらない夫**：
「一つ補足しておく。『迷ったら広く』は
　**どちらにも本物の根拠があって迷ったとき**の破り方であって、
　なんでも多対多にしろという意味ではない。
　多重度を広げるたびに『主はどれか』という新しい問いが生まれ、
　部分一意インデックスのような追加の制約が要る──今日見た通りだ。
　自由度はタダではない。だから **賭けである** と言った。

　次は連絡2、山田の遡及昇給だ。時間の幕に入る」

## 第3幕　原則 II ── 時間

### 3-1　UPDATEは歴史を殺す

---

**やらない夫**：
「連絡2を再掲する。
　『山田の基本給、4月にさかのぼって250,000円から265,000円に昇給。
　4月分と5月分の差額は6月に払います』。今は6月だ。どうする」

**やる夫**：
「UPDATEするお」

```sql
UPDATE employees SET base_salary = 265000 WHERE emp_id = 101;
```

「差額は (265000 − 250000) × 2ヶ月 = 30,000円を6月に足せばいいお」

**やらない夫**：
「その30,000円、どうやって計算した」

**やる夫**：
「新しい給与から古い給与を引いて……

　……あ。**古い給与、今UPDATEで消したお**。
　やる夫が暗算できたのは連絡メールに250,000って書いてあったからで、
　データベースはもう250,000を知らないお」

**やらない夫**：
「続けて攻撃する。半年後、労働基準監督署の調査が入って
　『4月分の賃金の計算根拠を示してください』と言われたら？
　賃金台帳は労働基準法で作成と保存が義務付けられている。
　保存期間は原則5年（当分の間は経過措置で3年）だ[^4]」

**やる夫**：
「4月時点の基本給が要る……けどもう無いお。
　しかも保存義務って **法律** かお。
　『遡及なんて滅多にない』とかいう確率の話じゃなくて、
　守れなかったら即アウトの **確定した制約** じゃないかお」

**やらない夫**：
「そうだ。第2幕の口座は『起きるかどうか』の賭けだったが、
　こちらは賭けですらない。**法令・監査要件は発生率0.1%でも設計要件だ**。
　確率の問題と義務の問題を混ぜるな。

　ではどう直す」

### 3-2　変更ログ ── 復元はできるが照会ができない

---

**やる夫**：
「変更の記録を別テーブルに残せばいいお。よくある監査ログだお」

```sql
CREATE TABLE salary_change_log (
    emp_id     integer,
    old_amount numeric,
    new_amount numeric,
    changed_at timestamptz NOT NULL,
    changed_by text NOT NULL
);
```

「本体の `base_salary` はUPDATEして、変わるたびにログを1行足す。
　これで『いつ、誰が、いくらからいくらに』は全部残るお」

**やらない夫**：
「では判定してもらおう。そのログを使って
　**『4月1日時点の、全社員の基本給一覧』** を出すSQLを書け。
　給与計算のやり直しに毎回必要になる、ごく普通の照会だ」

**やる夫**：
「えーと……各社員について『4月1日以前の最後の変更』の new_amount を拾う……」

```sql
SELECT e.emp_id,
       (SELECT l.new_amount
          FROM salary_change_log l
         WHERE l.emp_id = e.emp_id
           AND l.changed_at <= '2026-04-01'
         ORDER BY l.changed_at DESC
         LIMIT 1) AS salary_at_0401
FROM employees e;
```

「相関サブクエリで書けた……けど、待つお。
　入社してから一度も昇給してない社員は **ログに1行もない** から、
　この式だとNULLになるお。
　初任給をログに入れるには『入社した』という行が要って……
　あれ、それって『変更ログ』じゃなくて『出来事の記録』だお？」

**やらない夫**：
「その違和感は正しいから、いったん抱えたまま進め。
　もう一つの欠陥も言っておく。このログは
　**復元**（頑張れば当時の値を組み立てられる）はできるが、
　**照会**（過去のある時点を1発のWHEREで切り出す）に向かない。
　全社員×全時点の照会が毎回『最後の変更を探す』折り畳み計算になる。
　監査ログは『誰がいつ触ったか』を答える道具としては優秀だが、
　『当時いくらだったか』を日常業務で答える道具ではない」

### 3-3　イベントソーシング ── 美しすぎる答え

---

**やる夫**：
「さっきの違和感を最後まで進めていいかお。
　『変更ログが本体の添え物』だから中途半端なんだお。**逆にする**お。

　**起きたこと（イベント）を記録の本体にして、現在の状態は計算で導く**お」

```sql
CREATE TABLE events (
    event_id   bigint PRIMARY KEY,
    emp_id     integer,
    event_type text,      -- '入社' '昇給' '異動' '退職' ...
    payload    jsonb,     -- {"base_salary": 265000, "effective": "2026-04-01"}
    recorded_at timestamptz NOT NULL
);
```

「入社も昇給も異動も、全部このイベント列に追記する。
　`employees.base_salary` みたいな『今の値』はテーブルに持たない。
　欲しければイベントを頭から再生すれば出てくるお。

　これ、綺麗だお。第1幕の保留にしてた問い──
　『状態を入れるのか、出来事を入れるのか』──の答えは **出来事** だお！
　出来事さえ全部あれば、どの時点の状態でも再生できる。
　UPDATEもDELETEも要らない、追記だけの世界。
　監査ログはタダで手に入るし、賃金台帳の5年保存も自動でクリアだお。
　**これが真実の形だお！**」

**やらない夫**：
「良い着想だし、名前もついている。**イベントソーシング** だ[^5]。
　そして先例は数百年ある。会計の帳簿だ。
　複式簿記の仕訳帳は取引という出来事の追記だけでできていて、
　残高という状態は仕訳の集計で導かれる[^6]。
　帳簿を消しゴムで直すのは会計では犯罪で、間違いは反対仕訳の **追記** で直す。
　お前の直観は経理部が数百年やってきたことと同型だ。

　──その上で、だ。心酔したところを悪いが、反例をやる。
　カモメ製作所の親会社を含むグループ全体、**3,000人** にこの設計を適用したとする。
　毎月25日の給与計算で最初に必要になる
　**『全社員の今月の基本給一覧』** を出してみろ」

**やる夫**：
「3,000人ぶん、各人のイベントを頭から再生して……
　1人あたり入社から10年なら昇給・異動・改姓で数十〜数百イベント、
　全体で数十万イベントを毎回畳み込む……。

　SQLで書くと……event_type ごとに payload の中身も意味も違うから、
　『最新の昇給イベント』だけ拾えばいいわけでもなくて
　（途中に『降格』や『雇用形態変更』があったら？）、
　結局アプリでループ回して再生することになるお。
　**月給一覧という一番ありふれた照会が、テーブル1個舐めるだけで済まなくなる**お。

　ぐぬぬ……速度だけなら再生結果のキャッシュを持てば……
　あれ、でもそれって『現在の状態テーブル』の復活だお。
　しかも今度はイベントとキャッシュの **二重管理** になるお」

**やらない夫**：
「もう一発。5年運用した後、
　『昇給イベントのpayloadの形を変えたい』となったら？」

**やる夫**：
「過去の全イベントは古い形のまま残ってる……。
　再生ロジックが新旧両方の形を永遠に読めないといけないお。
　状態テーブルならALTER TABLE一発とデータ移行で済む話が、
　**過去の全イベントの解釈責任を未来のコードが背負い続ける** ことになるお。

　……美しかったのに。理論は完璧なのに、
　照会と運用が全部『再生』の向こう側に行っちゃうんだお」

**やらない夫**：
「棄却するか？」

**やる夫**：
「……全部は棄却したくないお。
　『事実を消さない、追記で直す』という **魂の部分** は
　監査と遡及の問題を根本から解いてたお。
　死んだのは『状態を一切持たない』という **原理主義の部分** だお。
　魂だけ残して、照会に耐える形に妥協できないかお」

**やらない夫**：
「その妥協点は既にある。第1幕の保留の答えの前半、
　『状態か出来事か』の実務の答えを言っておくと──**両方だ**。
　どう両立させるかの後半は第7幕でお前が見つける。保留を継続しろ。

　今は照会に耐える形を作る。ヒントは連絡2の文面そのものだ。
　『4月に **さかのぼって** 265,000円』──この文が主張している事実の形は何だ」

### 3-4　有効期間 ── 事実に賞味期限をつける

---

**やる夫**：
「『山田の基本給は265,000円である。**ただし4月1日以降について**』……

　あ。事実に **期間がくっついてる** んだお。
　『山田の基本給は250,000円（2023年10月1日から2026年4月1日まで）』
　『山田の基本給は265,000円（2026年4月1日から当分）』
　──1行1事実の『事実』に有効期間を含めれば、行は増えるだけで消えないお」

```sql
CREATE TABLE salaries (
    emp_id     integer REFERENCES employees,
    amount     numeric NOT NULL,
    valid_from date NOT NULL,
    valid_to   date NOT NULL DEFAULT 'infinity',
    PRIMARY KEY (emp_id, valid_from)
);

INSERT INTO salaries VALUES (101, 250000, '2023-10-01', '2026-04-01');
INSERT INTO salaries VALUES (101, 265000, '2026-04-01', 'infinity');
```

**やらない夫**：
「では判定してもらおう。さっき監査ログで地獄だった
　『4月1日時点の全社員の基本給一覧』は」

**やる夫**：
「WHEREを1回書くだけだお！」

```sql
SELECT emp_id, amount
  FROM salaries
 WHERE valid_from <= DATE '2026-04-01'
   AND DATE '2026-04-01' < valid_to;
```

「相関サブクエリも再生ループも消えたお。**照会、合格**だお。
　『現在の給与』も『今日』を入れるだけで同じ式。
　なんなら日常用のビューにしておくお」

```sql
CREATE VIEW current_salaries AS
    SELECT emp_id, amount FROM salaries
     WHERE valid_from <= current_date AND current_date < valid_to;
```

**やらない夫**：
「2点、細部を詰めさせてもらう。
　第一。境界の4月1日、お前のWHEREは
　`valid_from <=` と `< valid_to`──**始点は含み、終点は含まない** 半開区間だ。
　なぜ両端を含む閉区間にしなかった」

**やる夫**：
「無意識だったお……閉区間だと何が起きるか試すお。
　旧行の終わりを3月31日、新行の始まりを4月1日にした閉区間だと……
　うん、日付ならこれでも隙間なく繋がる、ように見えるお。

　……いや、駄目だお。2つ困るお。
　まず、後で粒度を時刻に細かくしたら3月31日23時59分59秒の後に隙間が生まれるお。
　それに『隣り合ってるか』の判定が
　**『前の終点の翌日が次の始点』** っていう計算入りの条件になるお。
　半開区間なら **『前の終点＝次の始点』** の単純比較で、
　どの時点もちょうど1行にだけ属する。半開のほうが事故りにくいお」

**やらない夫**：
「半開区間 `[from, to)` は期間データの定石だ。
　第二。この設計を攻撃する。次の2行を入れるぞ」

```sql
INSERT INTO salaries VALUES (101, 265000, '2026-04-01', 'infinity');
INSERT INTO salaries VALUES (101, 270000, '2026-09-01', 'infinity');
```

**やる夫**：
「9月に再昇給……あっ、**期間が重なってるお**！
　4月の行の valid_to を '2026-09-01' に閉じ忘れてるから、
　9月以降の給与を聞くと **2行返ってくる**お。
　山田の給与が同時に265,000円かつ270,000円。矛盾した事実が同居してるお。

　主キーは `(emp_id, valid_from)` だから素通しだお。
　始点が違えば別行扱い……重なりは主キーの射程の外だお。
　これ、第2幕の武器庫の表で言う『宣言的制約では不可』の枠かお？」

**やらない夫**：
「実は届く。PostgreSQLの **排他制約** だ。
　UNIQUEが『等しい行の共存禁止』なら、排他制約はその一般化で、
　『指定した関係が成り立つ行の共存禁止』を宣言できる[^7]」

```sql
CREATE EXTENSION btree_gist;  -- 等値と範囲を混ぜるための拡張

ALTER TABLE salaries ADD CONSTRAINT no_overlap
    EXCLUDE USING gist (
        emp_id WITH =,                              -- 同じ社員で
        daterange(valid_from, valid_to) WITH &&     -- 期間が重なる(&&)行は
    );                                              -- 共存できない
```

**やる夫**：
「『同じ社員で期間が重なる2行は存在できない』を宣言で書けるのかお！
　検収するお。

　閉じた旧行 `[4/1, 9/1)` ＋ 新行 `[9/1, ∞)` → 端が接するだけで重ならない、入る。✓
　閉じ忘れ旧行 `[4/1, ∞)` ＋ 新行 `[9/1, ∞)` → **制約違反で弾かれる**。✓

　合格・不合格の両方、期待通りだお。
　これで『どの時点を聞いても答えはちょうど1行』が **データベースの保証** になったお」

**やらない夫**：
「命名しておく。お前が組んだ
　『事実＋有効期間、追記で歴史を積む』形は、
　データウェアハウスの世界で **緩やかに変化する次元（SCD）のタイプ2** と
　呼ばれてきた定石だ[^8]。
　また SQL:2011 で標準SQLにも temporal table として取り込まれている[^9]。
　お前は30年もののパターンを再発明した──良いことだ。
　自力で掘った奴だけが、いつ使わないべきかも分かる」

### 3-5　二種類の時間 ── 世界の時計とデータベースの時計

---

**やらない夫**：
「仕上げに、時間の底にある区別を見せる。
　遡及昇給をもう一度よく見ろ。
　昇給が **効く** のは4月1日。カモメ製作所がそれを **知らせてきた** のは6月。
　この2つの日付は同じ種類の時間か？」

**やる夫**：
「……違うお。
　4月1日は **現実の世界で事実が変わった時刻**。
　6月は **データベースがそれを知った時刻**。
　今までの `valid_from` は前者しか記録してないお」

**やらない夫**：
「では意地悪な問いだ。
　5月25日、お前は5月分の給与を250,000円で計算して振り込んだ。
　遡及を知らなかったんだから当然だ。
　半年後、監査人がこう聞く。
　**『5月25日の計算時点で、データベースは山田の5月の給与を
　いくらだと認識していましたか』**」

**やる夫**：
「うぐ……。今の `salaries` を見ると、5月は265,000円の期間に入ってるお。
　でも当時のデータベースは250,000円と答えたはずだお。
　6月に遡及行を入れた瞬間、
　**『当時こう見えていた』という歴史が上書きされてる** ……

　あーーー！　これ、**同じ病気の再発だお！！**
　3-1でUPDATEが歴史を殺したのと同じことを、
　やる夫は『有効期間の行の書き換え』で一段上のレイヤーでやってるんだお！」

**やらない夫**：
「その通り。完全に対処するなら、時間軸を2本持つ。
　**有効時間**（現実世界でいつからいつまで真実か）と
　**システム時間**（データベースがいつからいつまでそう信じていたか）。
　両方を持つ設計を **バイテンポラル** と呼ぶ[^9]。
　1行の事実が『4月から有効、ただしDBがそう知ったのは6月25日以降』のように
　2次元の座標を持つ。
　SQL:2011のsystem-versioned tableはシステム時間側を自動化する仕組みだ」

**やる夫**：
「2次元……強力そうだけど、行の管理が明らかに重いお。
　全テーブルこれにするのかお？」

**やらない夫**：
「するな。判断基準はもう第2幕で手に入れているはずだ。
　この重さを払うべきかどうか、何で決める」

**やる夫**：
「賭けと義務の区別だお。
　**『答える義務のある過去の質問』のリストから逆算する**お。
　『4月時点の給与は？』に答える義務がある → 有効期間は必須。
　『5月25日時点で4月給与をどう認識していたか？』に答える義務が
　本当にあるテーブルだけ、バイテンポラルの重さを払う。
　義務のない質問のために設計を重くしないお」

**やらない夫**：
「そして給与に限っては、実はもっとずるい安上がりな手がある。
　『5月25日に計算した結果』そのものに関わる手だが──これは第7幕で自分で見つけろ。
　保留が2本になったな。

　最後に小ネタを一つ。世間のテーブルにはよく `deleted_flag` という列が生えている。
　消したことにするが行は残す、いわゆる論理削除だ。今日の目で見てどう思う」

**やる夫**：
「退職を `deleted_flag = true` にする、みたいなやつかお。
　……今日の目で見ると筋が悪いお。
　退職は削除じゃなくて **『在籍という事実の有効期間が終わった』** だお。
　`valid_to` に終端が入るだけで、行はもともと消さない設計なら
　フラグの出番自体がないお。
　『消したい』と思ったときは、たいてい
　『事実が終わった』か『そもそも間違いだった』かのどっちかで、
　その2つは区別して記録すべきだお」

**やらない夫**：
「では原則にまとめろ。二本目だ」

**やる夫**：
「**原則 II（時間）**：

　**事実は上書きしない。変化は新しい事実として追記するお。**
　**UPDATEとDELETEは『歴史はいらない』という宣言であって、**
　**その宣言をしていい列かどうかは業務と法律が決める。**
　**そして記録の形は『答える義務のある過去の質問』から逆算して選ぶお。**
　**現在だけ→上書き、時点の照会→有効期間、**
　**認識の歴史まで→バイテンポラル。義務がないなら重くしない。**

　……イベントソーシングの魂もここに入ってるお。
　あれは『事実は上書きしない』を原理主義まで振り切った姿だったんだお」

### 4-1　手当12種 ── JSONBの誘惑

---

**やらない夫**：
「カモメ製作所は好調で、従業員が30人から120人になった。
　人事制度も育って、手当が12種類に増えた。
　通勤、住宅、家族、資格、役職、寒冷地、夜勤、……。
　現行の設計は第1幕でお前が正規化した `employee_allowances`、
　1人1手当1行の縦持ちだ。現場から苦情が来ている。

　『社員1人の手当を見るのに、画面もSQLも12行相手にするのが面倒』
　『手当の種類を1個増やすたびにマスタ登録の手続きが要るのが面倒』」

**やる夫**：
「ふむ……最近のDBならこう書けるお。PostgreSQLには `jsonb` 型があるお[^10]」

```sql
ALTER TABLE employees ADD COLUMN allowances jsonb;

UPDATE employees SET allowances =
    '{"通勤": 12000, "住宅": 20000, "資格": 5000}' WHERE emp_id = 101;
```

「1人1行に戻って、手当は中に畳み込まれる。
　画面は1行取ればいいし、新しい手当はキーを増やすだけ。
　マスタ登録も要らない。**現場の苦情が両方消える**お。
　スキーマレス万歳だお」

**やらない夫**：
「では反例をやる。半年運用したこのデータで、
　『手当種別ごとの支給総額』──経理が毎月出す集計だ──を出せ」

```sql
-- 半年後の実データ（の一部）
-- emp_id=101: {"通勤": 12000, "住宅": 20000}

```

**やる夫**：
「集計は `jsonb_each_text` で中身を行に展開して……」

```sql
  FROM employees, jsonb_each_text(allowances)

```

「……結果を見て血の気が引いたお。

| 手当 | 総額 |
|--|--|

**やらない夫**：
「そうだ。ではJSONBは常に悪か？
　逆の例を出す。給与明細の画面レイアウト設定──
　『どの項目をどの順で表示するか、ユーザーごとのカスタマイズ』を
　保存したいとする。これも縦持ちの正規化テーブルにするか？」

**やる夫**：
「レイアウト設定かお……。

　**『DBがその中身を理解する必要があるか』** だお。
　集計・検索・制約・結合の対象＝DBが意味を理解すべきもの → 列と行に開く。

　行に開くのが正解。**やる夫のJSONB案は適用場所を間違えてた**んだお」

**やらない夫**：
「配列型や複合型も同じ物差しで測れる。

「CHECKやenumは種別の追加が **ALTER（スキーマ変更）** になる。

```sql

    type_name text NOT NULL UNIQUE,
    taxable   boolean NOT NULL     -- 課税対象か（通勤手当は非課税枠あり）

    emp_id     integer REFERENCES employees,
    type_id    integer REFERENCES allowance_types,

    valid_from date NOT NULL,

```

「あ、有効期間も最初からつけたお。

**やらない夫**：

### 5-2　再帰 ── SQLで木を歩く

---

**やらない夫**：
「実はSQLは1999年の標準化で木を歩く道具を手に入れている。
　**再帰共通テーブル式（WITH RECURSIVE）** だ[^12]。
　構造はこうだ──『種を1行置き、種から辿れる行を繰り返し足す』」

```sql
WITH RECURSIVE subtree AS (
    SELECT dept_id FROM departments WHERE dept_name = '営業本部'  -- 種
    UNION ALL
    SELECT d.dept_id                                              -- 子を足す
      FROM departments d JOIN subtree s ON d.parent_id = s.dept_id
)
SELECT SUM(cs.amount)
  FROM subtree st
  JOIN dept_assignments da ON da.dept_id = st.dept_id
  JOIN current_salaries cs ON cs.emp_id = da.emp_id;
```

**やる夫**：
「営業本部→その子→その孫→……を、**尽きるまで** 辿るのかお。
　深さ5になっても6になってもこのSQLは1文字も変わらない。合格だお。

　……ただ、『尽きるまで』が引っかかるお。尽きなかったらどうなるんだお」

**やらない夫**：
「いい嗅覚だ。では不合格例をやる。データ移行のミスでこうなっていたとする」

```sql
UPDATE departments SET parent_id = 21 WHERE dept_id = 10;
-- 第一営業部(10)の親が首都圏営業課(21)、
-- 首都圏営業課(21)の親は第一営業部(10)のまま
```

**やる夫**：
「10→21→10→21→……**円環だお！**
　再帰が永遠に終わらない。実行したら……無限ループで返ってこないお。

　そうか、`parent_id` は『各行に親は高々1つ』しか強制してないんだお。
　**木であること──円環がないこと──はデータベースは保証してくれない**。
　円環の禁止は行またぎどころか **経路またぎ** の性質で、
　第2幕の武器庫のどの宣言的制約よりさらに射程の外だお」

**やらない夫**：
「対処は二段構えだ。
　照会側は深さの上限ガードや循環検出（SQL標準にはCYCLE句もある）で
　**暴走しても死なない** ようにし、
　更新側は『親の付け替え時に祖先を辿って自分が出てきたら拒否』を
　トリガーかアプリで守って **円環を作らせない**。
　制約の射程外の不変条件は、こうやって運用系のコードが背負う。第2幕の続きだ。

　ここで、歴史の話をしておこう。
　実はリレーショナル以前のデータベースは、**データベース自体が木だった**。
　1960年代にアポロ計画の部品管理のために生まれたIBMのIMSは
　階層型データベースで、データは親子構造でしか持てなかった[^13]。
　コッドのリレーショナルモデルは『構造を決め打ちせず、
　照会のときに関係を組み立てる』というこの不自由さへの反逆だった。
　その反逆の子であるSQLが、木を歩くのだけは長年苦手で、
　再帰CTEを標準で手に入れたのが1999年──という皮肉な歴史だ」

**やる夫**：
「じゃあ1999年より前、というか再帰CTEが普及する前は、
　みんな木をどうやって照会してたんだお」

**やらない夫**：
「表を見る前に一つ発明しろ。部署Aの下にA1、その下にA11がある。
　各行へ文字列を一列だけ追加して、A配下を `LIKE` の前方一致一発で取れるようにしろ」

**やる夫**：
「自分までの道を `path` に持つお。
　Aは `'A'`、A1は `'A/A1'`、A11は `'A/A1/A11'`。
　A配下なら `WHERE path LIKE 'A/%'` で一発だお！　再帰が消えたお」

**やらない夫**：
「ではA1をBの下へ異動させる。更新は何行だ」

**やる夫**：
「A1だけじゃないお。A11以下、子孫全部の先頭を `'B/A1/...'` へ書き換える。
　……照会一発の代金を、**部分木まるごとの更新**で払ったお。
　前節のコード前方一致と同じで、文字列に構造を焼き込むと変更が刺さるんだお」

**やらない夫**：
「今お前が発明したものを含め、工夫の博物館を見せよう。
　どれも『再帰なしで子孫を1発で切り出す』ための発明だ。

| 方式 | 持ち方 | 子孫の照会 | 更新 |
|--|--|--|--|
| 隣接リスト | parent_id | 再帰が要る | 親の付け替え1行 |
| 経路列挙 | path = 'A/A1/A11' | 前方一致1発 | **配下全行のpath書き換え** |
| 入れ子集合 | 各部署に区間 [lft, rgt] | 区間の包含1発 | **挿入のたび大量の行を更新** |
| 閉包テーブル | 全祖先×子孫ペアを別表に | JOIN1発 | ペアの差し替え（行数は多い） |

　入れ子集合（nested sets）は、木を数直線上の入れ子の区間に写す発想で、
　『営業本部の区間に含まれる部署＝子孫』と照会できる。セルコが広めた[^14]。
　数学的には美しいが、部署を1個挿入すると区間を空けるために
　テーブルの半分の行のlft/rgtを更新する羽目になる」

**やる夫**：
「照会を1発にするために、構造の維持コストを払ってるんだお。
　……あれ、これ、どこかで見た構図だお。
　イベントソーシングの逆だお。あっちは記録が純粋で照会が地獄、
　こっちは照会が速くて更新が地獄。
　**どの操作を安くしたいかで形を選ぶ**、って話が形を変えてまた出てきたお」

**やらない夫**：
「よく繋げた。そして現代の答えはシンプルだ。

### 5-3　組織は毎年4月に死ぬ ── 木 × 時間

**やらない夫**：

**やる夫**：
「改編でparent_idをUPDATEして、第二営業部の行をDELETEしてたら……

```sql
CREATE TABLE departments (

    dept_name text NOT NULL,
    parent_id integer,

    valid_to   date NOT NULL DEFAULT 'infinity',
    PRIMARY KEY (dept_id, valid_from)
);

-- 第二営業部の統合消滅 = 行の有効期間が終わる（DELETEしない）
-- カモメ事業部の親変更 = 旧行を閉じて新行を追記

```

「照会するときは **『いつの組織図か』を必ず指定する**。

　『2027年3月31日時点の木』を丸ごと再現して歩けるお」

```sql
WITH RECURSIVE subtree AS (
    SELECT dept_id FROM departments
     WHERE dept_name = '営業本部'
       AND valid_from <= DATE '2027-03-31' AND DATE '2027-03-31' < valid_to
    UNION ALL
    SELECT d.dept_id
      FROM departments d JOIN subtree s ON d.parent_id = s.dept_id
     WHERE d.valid_from <= DATE '2027-03-31' AND DATE '2027-03-31' < d.valid_to
)
SELECT ...
```

**やらない夫**：
「所属（dept_assignments）のほうは？」

**やる夫**：

　……気づいたら、**第3幕の道具一式がそのまま道具箱から出てくる**お。
　有効期間、半開区間、排他制約、『いつの時点か』を聞く照会。

**やる夫**：

|--|--|--|

| 健康診断 | 会社が雇用主として保存する義務 → employments | |

**やらない夫**：
「それでいい。**分離の価値は、迷ったときに『どっちの事実か』という
　正しい問いを立てられるようになること** であって、

**やる夫**：
「山田さん、結婚して氏が変わったんだお（連絡4が来てたお）。

```sql
CREATE TABLE person_names (
```

**やらない夫**：
「その疑問は第7幕の獲物だから、取っておけ。いい位置に弾が溜まってきた。

### 6-2　識別子の材質 ── 連番、UUID、そして意味の毒

---

**やらない夫**：
「同一性の線を引くのがIDだと分かった以上、
　ID自体の作り方も設計の内だ。まず、買収の続報を伝える。
　アホウドリ重工グループの従業員番号は全グループで採番体系を統一する。
　**カモメ製作所の全員の従業員番号が変わる**」

**やる夫**：
「ぐ……でも今の設計なら耐えられるお。
　`emp_no`（従業員番号）は employments の **ただの属性** で、
　主キーは無意味な `employment_id` だから、
　番号の振り直しはUPDATE一発。外部キーは全部無傷だお。

　もし従業員番号を主キーにしてたら、
　全テーブルの外部キーを連鎖で張り替える大手術だったお。
　第5幕の部署コードと同じだお──
　**現実の識別子（名前・番号・コード）は、現実の都合で変わる**。
　変わるものを主キーにしたら、変化のたびにDB全体が揺れる。
　だから **内部の主キーは意味を持たない値（サロゲートキー）にして、
　業務上の番号は属性として持ち、UNIQUE制約で縛る** んだお」

**やらない夫**：
「では、その無意味な値をどう作る。一番簡単なのは？」

**やる夫**：
「**連番** だお。`GENERATED ALWAYS AS IDENTITY` で1, 2, 3……。
　短い、順序どおり、インデックスにも優しい。これでいいお」

**やらない夫**：
「内部に閉じたIDならそれで概ね正解だ。だが反例をやる。
　グループのEC子会社が、注文番号を連番のまま **顧客に見せていた**。
　競合他社が自分で1月に1回、6月に1回注文してみた。
　注文番号は 10452 と 19871 だった。競合は何を知る？」

**やる夫**：
「引き算するお。19871 − 10452 = 9,419。
　**5ヶ月で約9,400件、月間約1,900件の受注**……
　注文を2回しただけで **事業規模が丸見え** だお！！」

**やらない夫**：
「連番は『いくつ発行済みか』を運んでしまう。
　第二次大戦で連合軍がドイツ戦車の製造番号から
　生産台数を統計的に推定した話は German tank problem として有名だ[^16]。
　従業員番号も同じで、名刺の番号から採用規模が推測できる。
　**外部に見せる識別子に、発行数という意味が漏れている** ──
　部署コードに親子関係を焼き込んだのと同型の毒だ。対策は？」

**やる夫**：
「番号から情報を消せばいい……つまり **ランダムなID** だお。
　UUIDというやつかお。128ビットの乱数なら
　衝突もまず起きないし、順番も件数も読み取れないお」

**やらない夫**：
「ランダム型（UUIDv4）はその通り情報が漏れない。
　だが今度は逆側の税金を払う。
　主キーにはインデックス（B木）が張られる。
　連番なら新しい行は常に木の右端に追記されて、書き込みが局所に固まる。
　完全ランダムなUUIDv4だと挿入位置はどうなる」

**やる夫**：
「乱数だから木の **あちこちにバラバラに刺さる** お。
　ページのあちこちを触るからキャッシュに乗らないし、
　ページ分割も頻発する。挿入が多いテーブルだとじわじわ効いてくる税だお。
　……情報漏れを防ぐと局所性を失い、局所性を取ると情報が漏れる。
　どっちかしか選べないのかお？」

**やらない夫**：
「2024年に標準化された **UUIDv7** がその折衷だ[^17]。
　先頭48ビットがミリ秒のタイムスタンプ、残りが乱数。
　**時刻順に増える** から挿入は右端に固まりつつ、
　乱数部のおかげで発行数は数えられない。
　整理しておけ」

**やる夫**：
「表にするお。

| 方式 | 発行数・順序の漏れ | インデックス局所性 | サイズ |
|--|--|--|--|
| 連番 | **漏れる** | ◎ 右端追記 | 8バイト |
| UUIDv4（乱数） | 漏れない | **× バラバラ挿入** | 16バイト |
| UUIDv7（時刻+乱数） | 発行数は漏れない（時刻は載る） | ◎ ほぼ右端追記 | 16バイト |

　使い分けも言えるお。
　**内部に閉じるキーは連番で十分**。軽くて速い。
　**外部に見せる・複数システムで発行する識別子はUUIDv7を第一候補**、
　生成時刻すら見せたくないならv4。
　あと、連番には小ネタの罠もあったお──連番は歯抜けになる
　（ロールバックしても採番は戻らない）から、
　『番号が飛んでる！データが消えた！』と騒がないことだお」

**やらない夫**：
「最後に、識別子の中でも法律で挙動が決まっている特別枠を扱う。
　**マイナンバー** だ。全社員から集めることになる。
　`persons` の列として持って、照会にもJOINにもどんどん使う──でいいか？」

**やる夫**：
「これまでの流れだと『人の属性だからpersonsへ』……
　でも『特別枠』って前置きが怪しいお。何が特別なんだお」

**やらない夫**：
「マイナンバーは番号法により **利用目的が法定されている**。
　税と社会保障などの手続きのためにしか使えず、
　安全管理措置も義務付けられる[^18]。
　つまり『社内の名寄せに便利だから』という利用は違法だ。
　技術的にどう設計に反映する」

**やる夫**：
「利用できる場面が限られてるのに、personsに同居させたら
　人を照会するあらゆるコードの目に触れるお。
　だから **別テーブル（できればアクセス経路ごと分離）に隔離** して、
　外部キーでpersonsに繋ぎ、
　参照できる役割を絞って、暗号化して、アクセスを記録する。
　**JOINキーとして使うのは論外**──
　第一、マイナンバーで名寄せしていいのは法の許す手続きの中だけだお。
　同一性の線を引くのはあくまで自前の person_id で、
　外部の識別子は『対応を記録しておく属性』に留めるんだお」

**やらない夫**：
「では三本目、まとめろ」

**やる夫**：
「**原則 III（同一性）**：

　**『同じ』は観察できる事実じゃなくて、設計者が引く線だお。**
　**現実の1つの名前（従業員番号）の裏に、複数の同一性（人と雇用）が**
　**重なっていることがある。線が足りないと再入社で破綻する。**
　**そして引いた線は、意味を持たない不変の内部IDに固定するお。**
　**名前・番号・コードみたいな『意味のある識別子』は必ず変わるから、**
　**属性に降ろしてUNIQUEで縛る。**
　**外に見せるIDは、意味（発行数・順序・親子関係）が漏れない材質を選ぶお」**

**やらない夫**：
「三原則が揃った。残るは保留の最終回収──
　『状態か出来事か』の後半戦と、『明細の氏名は凍るべき』のお前の直感。
　仕上げの幕だ」

### 7-1　再計算したら金額が変わった

---

**やらない夫**：
「事件で始める。税務調査だ。調査官の依頼はシンプルで、
　**『2026年12月支給分の、全員の給与明細を出してください』**」

**やる夫**：
「余裕だお。うちのマスタは全部有効期間つきだお。
　基本給も手当も所属も『2026年12月時点』で切り出して、
　給与計算をもう一回回せば、当時と同じ明細が再現できる……

　……回したお。……**山田さんの手取りが、当時振り込んだ額と1,340円違う**お。
　どういうことだお!?　歴史は全部残してたのに！」

**やらない夫**：
「ズレの原因を調査しろ。容疑者は3人いる」

**やる夫**：
「調べたお……全員クロだったお。

　**容疑者1：マスタの遡及訂正**。1月末に『12月分の通勤経路の届け出が
　間違ってた』修正が入って、12月分の通勤手当の行が訂正されてた。
　今のマスタで再計算すると **訂正後** の額が出る。
　当時振り込んだのは **訂正前** の額だお。
　……これ、第3幕でやらない夫が言ってた
　『5月25日時点で何と認識していたか』問題の実物だお！

　**容疑者2：料率マスタ**。健康保険の料率が3月に改定されたとき、
　担当者（やる夫だお……）が料率テーブルを **上書きしてた**。
　料率は外部から与えられる数字だから油断してたけど、
　これも時間で変わる事実。同じ病気の3度目だお。有効期間、即つけるお。

　**容疑者3：計算ロジック**。2月にリリースした端数処理のバグ修正。
　今のプログラムは12月当時のプログラムと **挙動が違う**。
　当時の明細を再現するには当時のコードで走らせないといけない……」

**やらない夫**：
「容疑者1はバイテンポラルで解けるな。全マスタを2軸にするか？」

**やる夫**：
「うぐ……それでも容疑者3が残るお。
　データを2軸にしてもプログラムのバージョンは戻せない。
　古いバージョンを永久に動態保存？　現実的じゃないお。
　**『当時の計算を将来いつでも再演できる』という方針そのものが
　どこかで破綻してる** 気がするお……」

**やらない夫**：
「そこで問いを裏返せ。
　……そもそも、なぜ **再演** しようとしている？
　12月25日、給与計算が終わって、振込データを銀行に送った。
　その瞬間、計算結果は何になった？」

**やる夫**：
「振り込んだ瞬間……。
　！！　**出来事になったんだお！！**
　『山田に263,660円を支払った』はもう計算値じゃない。
　**実際に起きた、取り消せない事実** だお。
　事実なら原則IIの管轄だお──事実は記録する、上書きしない！

　つまり、**確定した瞬間に計算結果そのものを保存しておくべき** だったんだお。
　再演なんかしなくていい。当時の明細は当時の明細テーブルから出す。
　マスタがどう訂正されようが、プログラムがどう直ろうが、
　**支払ったという歴史は1ミリも動かない**」

```sql
CREATE TABLE payslips (                    -- 確定した支給：1人1回1行
    payslip_id   bigint PRIMARY KEY,
    employment_id integer NOT NULL REFERENCES employments,
    pay_month    date NOT NULL,            -- 2026-12-01
    paid_on      date NOT NULL,
    emp_no       text NOT NULL,            -- 当時の従業員番号を転記
    emp_name     text NOT NULL,            -- 当時の氏名を転記
    dept_name    text NOT NULL,            -- 当時の主務部署名を転記
    total_gross  numeric NOT NULL,
    total_deduction numeric NOT NULL,
    total_net    numeric NOT NULL
);

CREATE TABLE payslip_items (               -- 明細の各行も凍結
    payslip_id bigint REFERENCES payslips,
    item_name  text NOT NULL,              -- '基本給' '通勤手当' '健康保険料'...
    amount     numeric NOT NULL,
    PRIMARY KEY (payslip_id, item_name)
);
```

**やらない夫**：
「おい、`emp_name` と `dept_name`。氏名と部署名を **転記** したな？
　employment_id から辿ればいつでも引ける値だぞ。
　第1幕のお前なら『第3正規形違反！冗長！』と叫ぶところだが」

**やる夫**：
「叫ばないお。これは **わざと** だお。
　6-1で引っかかった『改姓したら過去の明細の氏名はどうなる』の答えがこれだお。
　明細は **発行された瞬間の世界の写し** で、
　山田さんが改姓しようが部署が改編で消えようが、
　2026年12月の明細には当時の氏名と当時の部署名が印字されてるべきだお。

　正規化が冗長を嫌うのは **『同じ事実を2箇所に書くと更新時にズレる』** からで、
　凍結した明細は **そもそも二度と更新しない**。
　更新しないものに更新時異常はないお。
　むしろここでpersonsをJOINして『今の氏名』が出るほうがバグだお」

**やらない夫**：
「**導出値は、確定した瞬間に事実へ昇格する**。
　正規化の教義ではなく、その値の性質が変わったから扱いが変わる──
　その理屈が言えるなら、もう非正規化を恐れる段階は卒業だ」

### 7-2　訂正は追記で ── 経理部が数百年やってきたこと

---

**やる夫**：
「でも待つお。凍結したら、**間違ってた場合** はどうするんだお。
　容疑者1の通勤手当、12月分は実際 **間違った額を振り込んでた** お。
　凍結した明細が間違ってる。直さないわけにはいかないお」

**やらない夫**：
「複式簿記の帳簿は、間違った仕訳を見つけたときどうすると言った？」

**やる夫**：
「消しゴム禁止で、反対仕訳を **追記** ……

　そうか、給与も同じだお。12月の明細はそのまま
　（間違った額を振り込んだのは事実だから、事実として保存）。
　**2月の明細に『12月分 通勤手当差額 +1,340円』という行を追記** するお。
　実務で『遡及支給』とか『調整』とか呼ばれてるあの明細行は、
　**反対仕訳と同じ発明** だったんだお[^6]。

　……それに、これでバイテンポラルの宿題も安く済むお。
　第3幕でやらない夫が言ってた『ずるい安上がりな手』ってこれだお？
　『5月25日時点で何と認識していたか』の答えは、
　**5月25日に確定した明細そのもの** が保存してる。
　認識の歴史が要るのは実務上ほぼ『計算に使った瞬間』だけだから、
　その瞬間を凍結しておけば、
　全マスタを2軸のバイテンポラルにしなくても監査に答えられるお」

**やらない夫**：
「そうだ。そして第1幕の保留──『状態か出来事か』──を、
　二段階目まで含めて回収しろ」

**やる夫**：
「回収するお。答えは **役割分担** だお。

　**変わりゆくもの（給与・所属・氏名・料率）は、状態＋有効期間の履歴** で持つ。
　照会が『WHERE 1発』で済む、第3幕で勝った形だお。

　**起きたこと（支給・確定・訂正）は、凍結された出来事** として追記する。
　こっちは二度と変わらないから、照会はさらに単純だお。

　イベントソーシングの魂は後者に生きてる。
　あの理論の間違いは出来事か状態かの **二者択一を迫った** ことで、
　実務の答えは『マスタは履歴つき状態、取引は出来事』の **両建て** だお。
　……経理部の帳簿が昔からその形だお。
　元帳とマスタ台帳、両方あるのが正解だったんだお」

### 7-3　締め ── 確定を状態機械で守る

---

**やらない夫**：
「仕上げに、確定の **周辺** を固める。3つ穴が残っている。

　穴1。給与計算バッチが30分走る間に、
　人事担当が山田の手当をUPDATEしたら、計算の前半と後半で違うマスタを見るぞ」

**やる夫**：
「計算の途中でデータが動く……。
　対策は2つ思いつくお。
　トランザクション分離レベルを上げて、
　バッチ全体が **開始時点のスナップショットだけを見る** ようにする
　（PostgreSQLならREPEATABLE READで一貫した読み取りになるお[^19]）。
　加えて業務側でも、**締め期間はマスタ編集をロック** する。
　『毎月20日で締め』というあの運用は、
　人間系の排他制御だったんだお」

**やらない夫**：
「穴2。『確定』した明細を、アプリのバグや操作ミスがUPDATEしてきたら？」

**やる夫**：
「『二度と更新しない』は今のところ **ただの決意** だお……。
　決意はDBの制約にするお。
　支給処理の状態を明示的に持って──計算中→確認中→**確定**──
　確定後の行はトリガーで変更を拒否する。
　ついでにアプリのDB接続ユーザーからUPDATE権限自体を剥がすお。
　防壁は宣言に近い層ほど固いお」

```sql
CREATE TABLE payroll_runs (               -- 計算の実行そのものを記録する
    run_id      bigint PRIMARY KEY,
    pay_month   date NOT NULL,
    status      text NOT NULL CHECK (status IN ('計算中','確認中','確定','取消')),
    as_of       timestamptz NOT NULL,     -- どの時点のマスタで計算したか
    executed_by text NOT NULL,
    program_ver text NOT NULL             -- 計算ロジックのバージョン
);
```

**やらない夫**：
「穴3は、いま自分で塞いだな。`payroll_runs`──」

**やる夫**：
「容疑者3（プログラムのバージョン）対策だお。
　当時のロジックを動態保存はできないけど、
　**『いつ、誰が、どの時点のマスタを、どのバージョンのプログラムで
　計算したか』という実行の事実** は記録できる。
　結果は明細に凍結済みだから、再演できなくても
　『この明細はこの条件の計算の産物』と説明はつくお。
　計算という行為自体も出来事だったんだお」

### 7-4　全部を要請から導き直す

---

**やらない夫**：
「では総仕上げだ。
　部品を継ぎ足してきた今の設計を、一度更地から導き直す。
　給与データベースが満たすべき要請を、番号をつけて列挙しろ」

**やる夫**：
「全幕から集めてくるお。

　**(1)** 遡及改定に耐える。過去の任意時点の給与・所属・料率を照会できる（第3幕）
　**(2)** 監査に答える。支払った内容は法定期間、不変のまま保存される（第3幕・第7幕）
　**(3)** 月次の給与計算・一覧が、単純なSQLで書ける（第3幕でESを棄却した理由）
　**(4)** 兼務・複数口座など、関係の多重度の例外に耐える（第2幕）
　**(5)** 雇用形態の違いと変更（登用）を、不正データなしに表せる（第4幕）
　**(6)** 再入社・改姓・番号変更で壊れない（第6幕）
　**(7)** 組織改編に耐え、当時の組織で集計できる（第5幕）」

**やらない夫**：
「候補となるアーキテクチャを大きい順に4つ挙げて、要請で殺していけ」

**やる夫**：
「やるお。

　**候補A：現在値だけの上書きモデル**（第1幕のやる夫）。
　→ (1)で即死。(2)も死ぬ。論外だお。

　**候補B：純イベントソーシング**（第3幕のやる夫）。
　→ (1)(2)は満点。だが **(3)で死ぬ**。
　月給一覧が再生計算になり、スキーマ進化の重荷も残る。

　**候補C：全テーブル・バイテンポラル**。
　→ (1)(2)(3)を満たすが、**過剰装備で自壊する**。
　全テーブル2軸の管理コストを払う根拠になる『答える義務のある質問』が、
　確定明細の凍結（7-1）でほぼ消える。義務なき重装備は要請違反じゃなくて
　**判断基準違反**（第3幕）だお。

　**候補D：ハイブリッド**。
　変わるものは **有効期間つき履歴**（salaries, dept_assignments,
　departments, person_names, 料率マスタ）、
　起きたことは **凍結された出来事**（payslips, payroll_runs）、
　操作の追跡は **監査ログ**、日常の照会は **currentビュー**。
　→ (1)〜(7)、全部通るお。

　……候補を並べて殺していったら、**残ったのは1個だけ** だお。
　しかもDの中身は、各幕で反例に殴られながら選んだ形そのものだお」

**やらない夫**：
「そうだ。この設計はお前の趣味でも流行でもなく、
　**要請を書き出せば、ほぼ一意に決まる**。
　逆に言えば、要請が違えば答えも変わる──
　遡及も監査もない使い捨ての社内ツールなら候補Aで正しいし、
　金融の約定履歴のように認識の歴史自体が商品なら候補Cが正解になる。
　設計の良し悪しは形にではなく、**要請と形の対応** に宿る。

　最終幕に行こう。答え合わせだ」

## 第8幕　答え合わせ ── お前が再発明したものの名前

---

**やらない夫**：
「物理の理論なら、最後は実験と照合する。
　設計に実験はないが、代わりの検証法がある。
　**先人が同じ崖から落ちて、同じ場所に橋を架けているかどうか** だ。
　お前が今日『発明』したものを、世間の名前と突き合わせてみろ」

**やる夫**：
「調べてきたお。……全部、名前がついてたお。

| やる夫が掘り当てた形 | 世間での名前 |
|--|--|
| 属性つき中間テーブル（第2幕） | ERモデルの関連（の属性）[^3] |
| 条件つき一意制約（第2幕） | 部分インデックス |
| 事実＋有効期間の履歴（第3幕） | SCDタイプ2[^8]、SQL:2011 application-time[^9] |
| 期間重複の禁止（第3幕） | 排他制約[^7] |
| 出来事を本体にする発想（第3幕） | イベントソーシング[^5] |
| 2本の時間軸（第3幕） | バイテンポラルデータモデル[^9] |
| タイプ別テーブルの3方式（第4幕） | 単一テーブル継承／クラステーブル継承／具象テーブル継承[^11] |
| 木を歩くSQL（第5幕） | 再帰CTE（SQL:1999）[^12] |
| 人と雇用の分離（第6幕） | Party と雇用の分析パターン[^15] |
| 意味を持たない不変キー（第6幕） | サロゲートキー |
| 時刻順で漏れないID（第6幕） | UUIDv7[^17] |
| 確定結果の凍結（第7幕） | スナップショット、読み取りモデルの永続化 |
| 訂正の追記（第7幕） | 反対仕訳[^6]、赤黒処理 |

　ちょっと悔しいけど、安心もしたお。
　やる夫が反例で殴られて曲がった場所、全部に先人の骨と橋があるお」

**やらない夫**：
「それが定石というものの正体だ──**先人の反例の化石** だ。
　名前を暗記した奴は霧の中でパターンを貼り間違えるが、
　崖から落ちて掘った奴は、どの崖にどの橋かを体が覚えている。

　では、最後の締めくくりだ。三原則を並べろ。
　今日の全行程を、他人に渡せる形で」

**やる夫**：
「並べるお。

　**原則 I（関係の自由度）**
　**多重度は観察ではなく、未来への賭け。**
　賭けは外れたときの痛みが小さい側──たいていは広い側──に張り、
　記録は広く、ビューと制約で狭く見せて、賭け金を減らす。

　**原則 II（時間）**
　**事実は上書きしない。変化は新しい事実として追記する。**
　記録の形（上書き／有効期間／2軸）は、
　答える義務のある過去の質問から逆算して選ぶ。

　**原則 III（同一性）**
　**『同じ』は事実ではなく、設計者が引く線。**
　1つの名前の裏に複数の同一性が重なっていないかを疑い、
　引いた線は意味を持たない不変のIDに固定する。

　それと、3つの原則の **使い方** も1枚にしておきたいお。
　どの幕でも、原則そのものより『どっちに倒すか』で迷ったから。

　・**痛みの非対称性で決める**：間違えたとき、移行手術か日常の税か。
　・**義務と確率を区別する**：法令・監査は0.1%でも設計要件。
　・**質問から逆算する**：将来答える義務のある照会のリストが仕様の本体。
　・**運用の隅を掘る**：例外は『ありえますか』ではなく
　　『そのときどうしてますか』で見つかる。Excelと備考欄が現場の本音。
　・**制約の射程を測る**：CHECKは行の中、UNIQUEは列集合、排他制約は行のペア、
　　行またぎ・経路またぎはトリガーとアプリが背負う。
　・**業務が決めることを技術で決めない**：端数処理、勤続の通算、みなし継続。
　　設計者の仕事は決断の肩代わりじゃなくて、どっちの決断も記録できる器だお」

**やらない夫**：
「振り返って気づくことはあるか」

**やる夫**：
「引っかかった場所が、全部要所だったお。
　手当1・手当2の列で正規化に勝った直後に、
　その正規化が3つの連絡に敗北するところから始まって──
　『dept_name2はさっきと同じ病気』『比率の合計はCHECKの射程外』
　『UPDATEが差額計算の根拠を消す』『イベントソーシングは月給一覧で死ぬ』
　『閉じ忘れの期間重複』『パートに等級5』『深さ5の組織で集計が黙って欠ける』
　『番号108が人と雇用を二重に背負ってた』『再計算したら1,340円ズレた』──
　単純な案が **なぜだめか** を1個ずつ数字で確かめたら、
　最後は要請からほぼ一意に設計が決まったお。

　あと、一番意外だったのはこれだお。
　物理の教科書と違って、設計には『正解の式』が最後に降ってこない。
　その代わり **『要請を書けば形が決まる』という手続きが正解** なんだお。
　未来が見えないのは最後まで変わらないけど、
　賭け方の作法は身についたお」

**やらない夫**：
「上出来だ。……ところで、報告がもう1件残っていてな。
　アホウドリ重工グループ、今度は自社ECサイトを始めるそうだ。
　『会員登録した顧客が実は従業員だった場合、ポイント優待を──』」

**やる夫**：
「従業員と顧客……どっちも **人** ……
　personsは給与DBの持ち物なのかお？　会社をまたいで人の線を引くのは誰だお？
　名寄せ？　それマイナンバー使えないやつだお？？」

**やらない夫**：
「それはマスターデータ管理という、また別の長い泥沼だ。
　今日はここまでにしよう」
