## 第2幕　並列化が壊れる日 ── モノイド

---

**やる夫**：

```go
func Fold[T, A any](xs []T, init A, op func(A, T) A) A {
    acc := init
    for _, x := range xs {
        acc = op(acc, x)
    }
    return acc
}

total := Fold(sales, 0, func(acc, s int) int { return acc + s })
```

**やらない夫**：
で、そのバッチが遅いと言っていたな

**やる夫**：

スライスを4分割して、goroutineで別々に集計して、最後に足すお。

試したら、逐次でも並列でも同じ答えになったお。

**どんな Fold も同じ手で分割できる**。シンプルイズベストだお

**やらない夫**：
では判定してもらおう。次のバッチも同じ手で並列化しろ。
在庫の引き落とし処理だ。`op` は引き算、出庫記録は `[10, 3, 2]`、初期値 0 で
『出庫合計の符号反転』を作っている。

**やる夫**：

**やらない夫**：
4分割は大げさだから2分割でいい。前半 `[10, 3]`、後半 `[2]`。

**やる夫**：

前半ワーカー：(0 − 10) − 3 = −13。

**やる夫**：

ぐぬぬ……並列化していい演算とダメな演算があるんだお。
やる夫は境界線を知らずに全バッチを書き換えて回るところだったお

**やらない夫**：

**やる夫**：

……これ、名前を知ってるお。**結合法則** だお。群論の授業の最初に出てきたお。

**やらない夫**：

**やる夫**：
何もしてないんだから、結果に影響しない値を返すべきだお。
合計なら 0、積なら 1、文字列連結なら空文字列 ""。
つまり、どんな a に対しても e・a = a・e = a になる e ……

これも知ってるお。**単位元** だお！
結合法則と単位元。……あれ？　群論で習ったセットから **逆元だけ抜けてる** お

**やらない夫**：
そこに気づいたなら、定義する資格がある。
『結合法則を満たす二項演算と、その単位元を持つ集合』──
群から逆元の要求を外したこの構造を、数学では **モノイド** と呼ぶ。
逆元を要求しないから、群になれない演算も仲間にできる。
文字列連結に逆元はあるか？

**やる夫**：
"abc" に何を連結しても "" には戻れないお。逆元なしだお。

なるほど、**『混ぜたら戻せない』系の演算はモノイド止まり** なんだお。
でも集計処理って、まさに混ぜたら戻せない演算だらけだお。
群より弱いのに、仕事では群より出番が多いお

**やらない夫**：
いい観察だ。ではそれをGoのコードに落とせ。
お前の並列Foldが要求している契約を、インターフェースとして書け

**やる夫**：
契約は『単位元をくれ』と『結合法則を満たす演算をくれ』の2つだから……

```go
// Monoid は結合法則 Combine(Combine(a,b),c) == Combine(a,Combine(b,c)) と
// 単位元則 Combine(Empty(),a) == Combine(a,Empty()) == a を満たすこと。
type Monoid[T any] interface {
    Empty() T
    Combine(a, b T) T
}

func FoldPar[T any](m Monoid[T], chunks [][]T) T {
    partial := make([]T, len(chunks))
    var wg sync.WaitGroup
    for i, chunk := range chunks {
        wg.Add(1)
        go func(i int, chunk []T) {
            defer wg.Done()
            acc := m.Empty() // 空チャンクなら単位元がそのまま返る
            for _, x := range chunk {
                acc = m.Combine(acc, x)
            }
            partial[i] = acc
        }(i, chunk)
    }
    wg.Wait()
    acc := m.Empty()
    for _, p := range partial {
        acc = m.Combine(acc, p)
    }
    return acc
}
```

書けたお。でも気持ち悪いところが一つあるお。
結合法則と単位元則は **コメントでしか書けない**。
コンパイラは `Combine` の中身が引き算でも通しちゃうお

**やらない夫**：
重要な気持ち悪さだから覚えておけ。
**型は形を検査するが、法則は検査しない**。法則を守るのは人間の仕事だ。
だからこそ、どの法則を守れば何が保証されるのかを **名前付きで** 知っている必要がある。
お前が今日 `Monoid` という名前を得たのはそういう意味だ。
では手持ちの演算を仕分けしろ。判定表を作れ

**やる夫**：
やってみるお。

| 台となる型 | 演算 | 単位元 | 結合法則 | 判定 |
|--|--|--|--|--|
| int | + | 0 | ○ | **モノイド** |
| int | × | 1 | ○ | **モノイド** |
| int | − | なし（0−a ≠ a） | ×（さっき死んだ） | 不合格 |
| float64 | 平均 | なし | ×（さっき死んだ） | 不合格 |
| string | 連結 | "" | ○ | **モノイド** |
| []T | append | 空スライス | ○ | **モノイド** |
| int | max | ？？ | ○ | 保留 |

max で詰まったお。結合法則はいいけど、単位元がないお。
int64 なら math.MinInt64 を置く手はあるけど、
……文字列の max（辞書順で大きい方）だと最小の文字列なんて存在しないお

**やらない夫**：
単位元が足りないだけの惜しい演算を救済する、定番の貼り薬がある。
値の集合に『値がない』という特別な要素を1個足して、
max(なし, x) = x と定義してしまえ

**やる夫**：
『値がないかもしれない値』かお。……それ、Haskellの入門書で見たお。
Maybe とか Optional とかいうやつだお。
あの時は『nullの上品版』くらいにしか思ってなかったけど、
**モノイドの単位元を製造する装置** でもあるのかお

**やらない夫**：
そいつの正体は次の幕で暴く。その前に、やり残した宿題がある。
お前は判定表で引き算と平均に『不合格』の烙印を押した。
だがそれは、お前の仕事で **一番出てくる集計** だろう。
『モノイドじゃないから並列化できません』で本当に済ませるのか

**やる夫**：
うっ、痛いところを突かれたお。
売上の平均、在庫の増減、価格の変動幅……全部モノイドじゃない演算だお。
でも並列化できないなら、1億件を1コアで舐めるしかないお。詰んでるお

**やらない夫**：
詰んでいない。発想を1段ずらせ。
**壊れる演算そのものを畳もうとするから詰む。
答えを作るのに十分な材料だけを、モノイドで畳め **。
道具を1個用意しておく。第2幕の FoldPar に『畳む前に各要素を変換する』を足しただけだ

```go
// 各要素を f でモノイド M の値に変換してから、並列に畳む
func FoldMapPar[T, M any](m Monoid[M], f func(T) M, xs []T) M {
    // 分割して各チャンクを Combine、部分結果を Combine（骨格は FoldPar と同じ）
    ...
}

// Monoid をその都度 type で書くのは冗長なので、関数2つを包むアダプタも用意
type funcMonoid[T any] struct {
    empty   func() T
    combine func(a, b T) T
}
func (m funcMonoid[T]) Empty() T         { return m.empty() }
func (m funcMonoid[T]) Combine(a, b T) T { return m.combine(a, b) }

// 足し算のモノイド（第2幕の判定表で合格したやつ）
var sumMonoid = funcMonoid[int]{
    empty:   func() int { return 0 },
    combine: func(a, b int) int { return a + b },
}
```

まず一番易しい引き算からいく。初期在庫 100 から出庫 `[10, 3, 2]` を引いて最終在庫を出す。
お前が本当に計算したいものを、式で書け

**やる夫**：
最終在庫 = 100 − 10 − 3 − 2 = 100 − (10+3+2) だお。
……あっ。**引き算を畳むんじゃなくて、出庫を足し算で畳んで、最後に1回引けばいい** お！
足し算はモノイドだから並列化できる。引き算は後処理に追い出す

```go
total := FoldMapPar(sumMonoid, func(x int) int { return x }, issues) // 10+3+2=15
final := 100 - total                                                  // 引き算は最後に1回
```

**やらない夫**：
それが第一の型、**壊れる演算を後処理に追い出す**。
お前の群論の言葉で言えば、引き算は加法群の逆元を使う操作だ。
畳むのは群の演算（+）、逆元（−）は最後に1回効かせればいい。
では本丸の平均だ。同じ発想で、何を畳めばいい

**やる夫**：
平均 = 合計 ÷ 個数。合計はモノイド、個数もモノイド（1個ずつ数える）……
**(合計, 個数) のペアを畳めばいい** お！
ペアの足し算 (s₁,n₁)+(s₂,n₂) = (s₁+s₂, n₁+n₂)、単位元は (0,0)。
結合法則も単位元もある立派なモノイドだお。割り算は最後に1回

```go
type Avg struct{ Sum float64; Count int }
var avgMonoid = funcMonoid[Avg]{
    empty:   func() Avg { return Avg{0, 0} },
    combine: func(a, b Avg) Avg { return Avg{a.Sum + b.Sum, a.Count + b.Count} },
}
acc := FoldMapPar(avgMonoid, func(x float64) Avg { return Avg{x, 1} }, xs)
average := acc.Sum / float64(acc.Count) // 後処理で割る
```

**やらない夫**：
これが第二の、そして最重要の型だ。**累算器を答えより豊かにする**。
答えは数1個だが、畳む途中の器はペアだ。
平均そのものはモノイドをなさないが、平均の **材料** はモノイドをなす。
この一手で、さっき分割位置で答えが割れたバグ（2.25 だの 1.75 だの）は消える。
どう分割しても (合計,個数) は同じ値に畳まれるからな。
あと3つ、同じ発想で片付く。範囲、分散、最頻値だ。まず範囲

**やる夫**：
範囲 = 最大 − 最小 だお。max も min もモノイド（第2幕で見た）。
**(最小, 最大) のペアを畳んで、最後に引く** お。
引き算（型1）とペア（型2）の合わせ技だお

```go
type Range struct{ Min, Max int }
var rangeMonoid = funcMonoid[Range]{
    empty:   func() Range { return Range{math.MaxInt, math.MinInt} }, // 単位元
    combine: func(a, b Range) Range { return Range{min(a.Min, b.Min), max(a.Max, b.Max)} },
}
r := FoldMapPar(rangeMonoid, func(x int) Range { return Range{x, x} }, xs)
spread := r.Max - r.Min // 後処理
```

**やらない夫**：
Empty に MaxInt と MinInt を置くのがミソだ。
『min の単位元は +∞、max の単位元は −∞』──さっき max で困った単位元問題を、
ペアの各成分で解いている。次、分散。平均の親戚だ

**やる夫**：
分散 = 平均(x²) − 平均(x)² だったお。
平均を出すのに (合計, 個数) が要ったから、x² の合計も足せば……
**(Σx², Σx, 個数) の三つ組を畳む** お！ 最後に公式に入れる

```go
type Stats struct{ SumSq, Sum float64; N int }
var statsMonoid = funcMonoid[Stats]{
    empty:   func() Stats { return Stats{0, 0, 0} },
    combine: func(a, b Stats) Stats { return Stats{a.SumSq + b.SumSq, a.Sum + b.Sum, a.N + b.N} },
}
s := FoldMapPar(statsMonoid, func(x float64) Stats { return Stats{x * x, x, 1} }, xs)
mean := s.Sum / float64(s.N)
variance := s.SumSq/float64(s.N) - mean*mean // 1パスで分散
```

**やらない夫**：
これが実データベースや分散処理基盤が分散・標準偏差を **1パス** で出す方法そのものだ。
器を (2成分) から (3成分) に太らせただけ。もっと複雑な統計量も同じ手で1回の走査に収まる。
最後に、毛色の違うのを1つ。最頻値だ

**やる夫**：
最頻値 = 一番多く出た値。……数の足し算じゃ捕まえられないお。
でも『各値が何回出たか』を数えれば……
**値→回数のマップを畳む** お！ マップの併合（同じキーは回数を足す）はモノイドだお。
単位元は空マップ。最後にマップの中で回数最大のキーを選ぶ

```go
histMonoid := funcMonoid[map[T]int]{
    empty: func() map[T]int { return map[T]int{} },
    combine: func(a, b map[T]int) map[T]int {
        out := map[T]int{}
        for k, v := range a { out[k] += v }
        for k, v := range b { out[k] += v } // キーごとに回数を合流
        return out
    },
}
hist := FoldMapPar(histMonoid, func(x T) map[T]int { return map[T]int{x: 1} }, xs)
mode := argmaxKey(hist) // 後処理で最頻値を1個選ぶ
```

**やらない夫**：
ヒストグラムというモノイド。これで最頻値も並列化できた。
……だが正直に、限界も1つ教えておく。中央値はどうだ

(合計,個数) みたいな小さい器には畳めないお。
**全部の値、少なくとも全ヒストグラムを持っておくしかない** お。

**やらない夫**：


**やる夫**：

通知バッチなら『未登録者リストに載せる』のと『データ不整合のアラートを上げる』で
やることが違うのに、区別できないお。
しかも int のフィールドだったら 0 が **正当な値** かもしれないから、
ゼロ値作戦はもっと悲惨だお。
……『値がない』は、値の世界の中の1個の値じゃ表現できないんだお

**やらない夫**：
その言語化ができたなら、正しい道具は自分で作れる。
『T型の値があるかもしれないし、ないかもしれない』を、型として書け。
第2幕の最後にお前が言った『値がないかもしれない値』だ

**やる夫**：
Goのジェネリクスで書くお

```go
type Option[T any] struct {
    val T
    ok  bool
}

func Some[T any](v T) Option[T] { return Option[T]{val: v, ok: true} }
func None[T any]() Option[T]    { return Option[T]{} }

// 取り出すときは必ず「ない」場合と向き合わされる
func (o Option[T]) Get() (T, bool) { return o.val, o.ok }
```

ポインタと違って、**中身を使うには ok を見るしかない** 構造だから、
チェック忘れがコンパイル時に締め出されるお。
じゃあ社内APIも `FindEmployee(id) Option[Employee]` に直して……

あれ。でも待つお。これ、中身に触るたびに `Get()` して if を書くなら、
nilチェックピラミッドが okチェックピラミッドに変わっただけだお？

**やらない夫**：
そのとおり、入れ物だけでは何も解決しない。
解決するのは入れ物に付ける **操作** のほうだ。
まず一番単純な操作からいく。
『中身があれば関数を適用し、なければないまま』を書け

**やる夫**：
スライスの各要素に関数をかける Map と同じノリだお。
Option は『要素数が0か1のコンテナ』だと思えばいいお

```go
func MapOpt[T, U any](o Option[T], f func(T) U) Option[U] {
    if v, ok := o.Get(); ok {
        return Some(f(v))
    }
    return None[U]()
}
```

これで `MapOpt(FindEmployee(42), func(e Employee) string { return e.Name })` みたいに
if なしでつなげるお。よし、本丸の BossEmail を Map の連鎖で書き直すお。
上司を取る関数は `GetBoss(e Employee) Option[Employee]` になってるから……

```go
e := FindEmployee(id)             // Option[Employee]
b := MapOpt(e, GetBoss)           // Option[Option[Employee]] ← ！？
```

……コンパイルは通るけど、型がおかしいお。
**Option[Option[Employee]]** って何だお。オプショナルのオプショナルだお。
この上さらにメールを取ったら Option[Option[Option[string]]] だお。
層が1段ずつ増えて、最後に3段の okチェックをやらされるお。本末転倒だお

**やらない夫**：
なぜ二重になったのか、原因を言え

**やる夫**：
Map に渡す関数の約束は `T → U`、つまり **失敗しない関数** だお。
そこに `Employee → Option[Employee]` っていう **失敗するかもしれない関数** を
渡したから、Map が包む Option と、関数が自前で返す Option で二重包装になったお。
悪いのは Map じゃなくて、用途に合わない道具を使ったやる夫だお

**やらない夫**：
では要るものは何だ

**やる夫**：
二重包装を一重に潰す操作だお。名前は……Flatten でいいお

```go
func Flatten[T any](oo Option[Option[T]]) Option[T] {
    if o, ok := oo.Get(); ok {
        return o
    }
    return None[T]()
}
```

外が Some なら中身をそのまま返す、外が None なら None。
それで、Map してすぐ Flatten するのがワンセットになるなら、
最初から合体させた操作を作るお。名前は flatMap、HaskellやScalaで見たやつだお

```go
func FlatMap[T, U any](o Option[T], f func(T) Option[U]) Option[U] {
    if v, ok := o.Get(); ok {
        return f(v)
    }
    return None[U]()
}
```

これで書き直すと──

```go
func BossEmail(id int) Option[string] {
    return FlatMap(FlatMap(FindEmployee(id), GetBoss), GetEmail)
}
```

**3段のピラミッドが1行になったお！**
nilチェックは FlatMap の中に1回だけ書いてあって、使う側には二度と現れないお

**やらない夫**：
エラー理由も運びたければ、`ok bool` を `err error` に替えた
`Result[T]` を作れば同じ FlatMap がそのまま書ける。
失敗の種類が増えただけで、**形は完全に同じ** だ。
もう一つ。スライスの FlatMap も書いてみろ。
『社員 → その部下たち』を2段たどって『部下の部下』を全員リストアップする処理だ

**やる夫**：
Map だと `[][]Employee` になっちゃうから、appendで平らに潰しながら集めるお

```go
func FlatMapSlice[T, U any](xs []T, f func(T) []U) []U {
    var out []U
    for _, x := range xs {
        out = append(out, f(x)...)
    }
    return out
}

grandReports := FlatMapSlice(FlatMapSlice([]Employee{e}, Reports), Reports)
```

……いま書いた Option とスライス、それにさっきお前が言ってた Result も。
中身も用途も全然違うのに、
**Map があって、二重包装ができて、Flatten で潰して、FlatMap になる**。
同じ映画を何度も見せられてる気分だお。これは偶然じゃないお

**やらない夫**：
偶然ではない。だが名前を与えるのはまだ先だ。
先にこの幕で観察した事実を、法則の形で棚卸ししておく。
まず、Map を2回続けて呼ぶのと、合成した関数で1回呼ぶの、結果は同じか？

**やる夫**：
`MapOpt(MapOpt(o, f), g)` と `MapOpt(o, gとfの合成)` かお。
Some(v) なら両方 Some(g(f(v)))、None なら両方 None。**同じだお**。
スライス版でも同じだお。
実務的には大事な話で、1億件のスライスに Map を2回かけると
中間スライスが1個丸ごと余分に生まれるから、
やる夫はいつも1回にまとめるリファクタをやってるお。
……あれは **結果が変わらないと信じてるから** できるリファクタだったんだお

**やらない夫**：
その『信じてる』の根拠は第5幕で数学の名前がつく。棚卸しを続けろ。
FlatMap の連鎖はどうだ。`FlatMap(FlatMap(m, f), g)` と、
『f してから g する合成関数』を一発で FlatMap するのと、同じか？

**やる夫**：
BossEmail で言えば、
『(社員を引いて、上司を取る) してから、メールを取る』と
『社員を引いて、(上司を取ってからメールを取る)』……
途中で None が出たらどっちも None で落ちるし、
全部 Some ならどっちも同じ値に届く。**同じに見えるお**。
証明はできないけど、少なくとも Option では成り立ってそうだお

**やらない夫**：
よし。観察事実として記録しろ。ここまでの棚卸し：

**観察1**：Map は2回がけと合成1回がけが一致する（Option・スライスで確認）
**観察2**：結合法則＋単位元＝モノイドが並列集計を支えている（第2幕）
**観察3**：FlatMap の連鎖は括り方を変えても結果が変わらない **らしい**（未証明）

それから、お前はさっき見逃せないことを言った。
『f **してから** g する』と言ったな。
`GetBoss` の型は `Employee → Option[Employee]`、
`GetEmail` は `Employee → Option[string]`。
この2つを『してから』でつなぐのは、普通の関数合成 g∘f か？

**やる夫**：
……違うお。合成しようにも **型が食い違ってる** お。
GetBoss の出力は `Option[Employee]` で、GetEmail の入力は `Employee`。
包装1枚分ズレてて、そのままじゃ刺さらないお。
なのに FlatMap を挟むと、あたかも合成できてるかのように数珠つなぎになる。

……やる夫たちは一体 **何を** 合成してるんだお？

**やらない夫**：
この旅で最良の問いだ。だが答えはお前が第6幕で自分で見つける。
付箋を貼って保留にしておけ。

うっ……古傷だお

### 4-2　ログを流したら法則が要った ── Writerの自作

---

**やらない夫**：
題材はまた実務からもらう。第3幕の BossEmail パイプラインに
監査ログを仕込みたい。各ステップが『何をしたか』を文字列で残す。
ただし printf でその場で出力するのは禁止だ。
テストで検証できるように、ログは **値として** 呼び出し元に返せ

**やる夫**：
つまり各ステップの戻り値を『本来の値とログのペア』にするんだお

```go
type Writer[T any] struct {
    Val T
    Log string
}

func FindEmployeeW(id int) Writer[Employee] {
    e := findEmployee(id)
    return Writer[Employee]{e, fmt.Sprintf("find(%d);", id)}
}
```

で、これを連結すると……あっ、また地獄が始まったお。
次のステップに Val を渡して、返ってきたログを自分のログに **手で** つなげて……

```go
func BossEmailW(id int) Writer[string] {
    w1 := FindEmployeeW(id)
    w2 := GetBossW(w1.Val)
    w3 := GetEmailW(w2.Val)
    return Writer[string]{w3.Val, w1.Log + w2.Log + w3.Log} // 手作業！
}
```

ログのつなぎ忘れ、順序間違い、絶対やらかすお。
……いや、待つお。この形、見覚えがあるお。
『値を次に渡して、オマケ（ログ）は裏で合流させる』──
**FlatMap の仕事** だお！　Writer 用の FlatMap を書けばいいお！

```go
func FlatMapW[T, U any](w Writer[T], f func(T) Writer[U]) Writer[U] {
    r := f(w.Val)
    return Writer[U]{r.Val, w.Log + r.Log} // ログは自動で合流
}

func BossEmailW(id int) Writer[string] {
    return FlatMapW(FlatMapW(FindEmployeeW(id), GetBossW), GetEmailW)
}
```

**やらない夫**：
Option のときと同じ1行に戻ったな。
ではもう一つ道具を足す。『ログを出さずに、ただの値を Writer の世界に持ち込む』
関数を書け。パイプラインの先頭で要るはずだ

**やる夫**：
値をくるむだけの関数だお。ログは……何も起きてないんだから空文字列だお

```go
func PureW[T any](v T) Writer[T] { return Writer[T]{v, ""} }
```

……ん？　今、無意識に "" を選んだけど、
これ **選ばされてた** お。
もし PureW のログを "start;" とかにしたら、
パイプラインのどこに PureW を挟むかでログが変わっちゃう。

### 4-3　Goで書けない一行 ── 高階型の壁

```go
type Monad[M[_] any, T any] interface {   // M[_] ← こんな型パラメータはない
    FlatMap(f func(T) M[U]) M[U]          // U もメソッドに導入できない
}
```

Goの型パラメータに渡せるのは int とか Option[int] みたいな **完成品の型** だけで、

Haskellで同じものを書いてみろ。かじった記憶で構わん

**やる夫**：
確かこう書くお

```haskell
class Monad m where
  return :: a -> m a
  (>>=)  :: m a -> (a -> m b) -> m b

instance Monad Maybe where
  return = Just
  Nothing >>= _ = Nothing
  Just x  >>= f = f x
```

……あっ。`class Monad m` の **m が、作りかけの型のまま変数になってる** お。
`m a` って書けてる。Goで書けなかった ??? の行が、ここでは普通に書けてるお。

**どのモナドでも動く汎用関数** が書けるんだお。入門書に出てきた謎関数の正体だお

**やらない夫**：
整理しよう。この幕で分かったこと：

・Writer の3法則はログの **モノイド則** に支えられている（2点セット、2回目）
・『モナド一般』を型で語るには高階型が要る。Goは個別実装まで、共通化はHaskellから

そして本題はここからだ。
『型コンストラクタを変数として扱い、法則付きの構造を一般に語る』──
Haskellの型クラスがやっているこれを、
**100年分の一般性で** やるために作られた数学がある。それが圏論だ。
次の幕から、その道具を一個ずつお前に再発明してもらう

### 5-2　世界の写し絵 ── 関手

第4幕でさんざん苦しんだ『作りかけの型』、
**型を1個食べて型を返すやつ** だお。

int ↦ Option[int]、string ↦ Option[string]、って感じで

**やらない夫**：

『対象を対象に写す対応』だけあれば、Option の重要な性質は語り尽くせるか？

**やる夫**：
語り尽くせる……と言いたいところだけど、引っかかるお。
第3幕でOptionが便利になったのは `MapOpt` を作ってからだお。
つまり対象の対応だけじゃなくて、**射も一緒に写してる** お。
`f: int → string` を `MapOpt(_, f): Option[int] → Option[string]` に写す。

**やらない夫**：
ではペアなら何でもいいのか。反例をやる。
スライスで、正規の Map と型がまったく同じ、こういう写し方を考えろ

```go
// 邪悪なMap：適用する前にスライスを逆順にする
func EvilMap[T, U any](xs []T, f func(T) U) []U {
    out := make([]U, len(xs))
    for i, x := range xs {
        out[len(xs)-1-i] = f(x)
    }
    return out
}
```

型シグネチャは正規の Map と区別がつかない。
これを『スライスの射の写し』に採用してよいか。ダメなら、ダメな理由を **法則の形で** 言え

**やる夫**：
直観的には論外だお。でも法則の形で言えって言われると……。
うーん、EvilMap の何が嫌かというと、**写す前に余計なこと** をしてるところだお。
『何もしない関数』を写したら『何もしない』になってほしいのに、
`EvilMap(xs, 恒等関数)` は逆順スライスを返す。恒等が恒等に写らない。

それと観察1も壊れるお。
`EvilMap(EvilMap(xs, f), g)` は逆順の逆順で元の順に戻るけど、
`EvilMap(xs, g∘f)` は1回だけ逆順。2回がけと合成1回がけが一致しない。

つまり欲しい法則はこの2つだお：
**(1) 恒等射は恒等射に写れ**
**(2) 合成の写しは、写しの合成と一致しろ**

**やらない夫**：
その2法則を添えた『圏から圏への写し絵』を **関手（functor）** と呼ぶ。

$$F(\mathrm{id}_A) = \mathrm{id}_{F(A)}, \qquad F(g \circ f) = F(g) \circ F(f)$$

対象の対応＋射の対応＋この2法則。それが関手のすべてだ。
Option もスライスも Writer も、型の圏から型の圏への関手。
特に **出発点と行き先が同じ圏** である関手を **自己関手（endofunctor）** と呼ぶ

**やる夫**：
自己関手ってお前だったのかお！！　呪文の第一容疑者！！
『自己関手』なんて仰々しい名前だから宇宙的な何かかと思ったら、
**Option とかスライスとか、やる夫が毎日使ってる型コンストラクタ（+Map+法則）**
のことだったお！

**やらない夫**：
呪文の解読、3分の1完了だな。
実利の話もしておく。お前が第3幕で言った
『Map 2回を1回にまとめるリファクタ』──1億件のスライスなら
中間スライス1本と1億回のループが丸ごと消えるやつだ。
あの書き換えの合法性を保証しているのが関手則(2)そのものだ。
GHCのような処理系は、この種の融合変換を最適化として自動でやる。
**法則は、人間とコンパイラの両方にとっての書き換え許可証** だ

**やる夫**：
観察1、回収されたお。
『信じてるからできるリファクタ』が『法則が保証するリファクタ』に格上げだお。
Haskellで書くとこうだったお

```haskell
class Functor f where
  fmap :: (a -> b) -> f a -> f b
-- 法則（コードでは書けない、Goのときと同じ事情だお）
-- fmap id == id
-- fmap (g . f) == fmap g . fmap f
```

**やらない夫**：
そうだ。そして型クラスの宣言に法則がコメントでしか書けないのは
Haskellも同じだという点は、公平のために言っておく。
法則を型システムで機械検査したければ、Agda や Coq のような
定理証明系の領分になる。それは国境の向こう側の話だ

### 5-3　型を覗かない関数 ── 自然変換

---

**やらない夫**：
道具箱の最後の一個だ。これで第6幕の解読作業に必要な装備が揃う。
題材：お前のコードベースに、こういう便利関数はないか

```go
// スライスの先頭要素。空なら None
func HeadOption[T any](xs []T) Option[T] {
    if len(xs) == 0 {
        return None[T]()
    }
    return Some(xs[0])
}
```

**やる夫**：
あるお。むしろ書いたお。
スライスをそのまま触ると index out of range が怖いから、
先頭を Option にくるんで返すやつだお

**やらない夫**：
この関数の圏論的な身分を考える。
スライスも Option も自己関手だと分かった。
`HeadOption` は **スライスという関手から Option という関手への乗り換え** だ。
しかも任意の型 T で一様に動く。
ここで問題。『任意の型で動く関数なら何でも、まっとうな乗り換えと言えるか』

**やる夫**：
型パラメータで書けてる時点でまっとうじゃないのかお？
……いや、その言い方をするってことは反例があるんだお。先に検査基準をくれお

**やらない夫**：
基準はお前に作らせる。
乗り換えと Map の **順序交換** を考えろ。
`[]int` から出発して `Option[string]` に行く経路は2本ある。
経路A：先に HeadOption で乗り換えて、あとで Option 側の Map で変換。
経路B：先にスライス側の Map で変換して、あとで HeadOption で乗り換え。
`xs = []int{1, 2, 3}`、f = 『10倍して文字列化』で両経路を計算しろ

**やる夫**：
経路A：HeadOption → Some(1)、Map(f) → **Some("10")** 。
経路B：Map(f) → ["10", "20", "30"]、HeadOption → **Some("10")** 。
一致だお。空スライスでも None と None で一致。
まあ、そりゃそうって感じだお。
HeadOption は『先頭を取る』っていう **位置の操作** しかしてなくて、
中身の値には指一本触れてないんだから、変換と順序を入れ替えても影響しようがないお

**やらない夫**：
その『そりゃそう』を裏切る関数を書いてやる。Goには型を覗く裏口があるからな

```go
// 邪悪な先頭取得：中身がintのときだけ0を返す
func CheatHead[T any](xs []T) Option[T] {
    if len(xs) == 0 {
        return None[T]()
    }
    var probe any = xs[0]
    if _, isInt := probe.(int); isInt {
        var zero T
        return Some(zero) // int のときだけ先頭を無視して零値！
    }
    return Some(xs[0])
}
```

同じ2経路を計算しろ

**やる夫**：
経路A：CheatHead([]int{1,2,3}) → 中身が int だから **Some(0)**、
f を Map して **Some("0")** 。
経路B：先に Map(f) → ["10","20","30"]、中身は string だから素直に **Some("10")** 。

**Some("0") ≠ Some("10")、経路がズレたお！！**
CheatHead は型パラメータの見た目は総称なのに、
中で **型を覗いて挙動を変えてる** から、変換の前と後で別人になるんだお

**やらない夫**：
そこで基準が立つ。関手 F から関手 G への乗り換え α がまっとうであるとは、
**すべての射 f について、この順序交換が成立すること**

$$G(f) \circ \alpha \;=\; \alpha \circ F(f)$$

これを **自然性**、この条件を満たす乗り換えを **自然変換** と呼ぶ。
HeadOption は自然変換。CheatHead は自然変換ではない。
合格例と不合格例、両方お前が自分の手で判定したことになるな

**やる夫**：
『自然』って、ふわっとした褒め言葉だと思ってたら、
**両経路一致という機械的に検査できる等式** だったのかお。
……ということは、だお。Haskellには型を覗く裏口が（普通に書く限り）ないお。
型switchもreflectionもない。
ってことは、Haskellで `[a] -> Maybe a` って型の関数を書いたら、
**何を書いても自動的に自然変換になる** んじゃないかお？

**やらない夫**：
鋭い。それは実際に定理になっていて、**パラメトリシティ** と呼ぶ。
『型シグネチャだけから、実装を見ずに、その関数が満たす等式が導ける』
──ワドラーが1989年に **Theorems for free!（タダで定理を）** という
題の論文にしている[^9]。
識者が『多相関数は自然変換』と言うときの根拠がこれだ。
そしてこの逆読みが実務でも効く。
`fmap f . headOption = headOption . fmap f` の右辺と左辺、
1億件のスライスならどちらで計算したい？

**やる夫**：
左辺だお。先に先頭だけ取れば f の適用は最大1回、
右辺だと1億回 Map してからほぼ全部捨てるお。
**自然性は『重い処理を後ろから前に動かしていい』許可証** でもあるんだお。
データベースの述語プッシュダウンと同じ匂いがするお

**やらない夫**：
では第5幕の武器庫を棚卸しして締める。

・**圏**：対象と射、合成と恒等射、結合法則＋単位元則。『合成がまともな世界』の基準
・**モノイド**＝対象1個の圏。お前の Monoid インターフェースは圏の特殊例
・**関手**：圏から圏への写し絵。対象の対応＋射の対応＋2法則。Map融合の許可証
・**自己関手**：型の圏から型の圏への関手。Option・スライス・Writer の正体
・**自然変換**：関手から関手への乗り換えで、すべての Map と順序交換できるもの

歴史の注釈を一つ。この3点セットは1945年にアイレンベルクとマクレーンが
一本の論文で導入した[^10]。動機はお前と同じ逆算だ。
彼らは『自然な対応』という言い回しを厳密にしたくて自然変換を定義し、
自然変換を定義するために関手が要り、関手を定義するために圏が要った。
**圏論は、圏が偉いから始まったんじゃない。『自然』を定義したくて始まった**

**やる夫**：
発明の順番が教科書の章立ての逆なのかお。
やる夫の旅も同じだお。Map の融合や HeadOption の順序交換が先にあって、
それを言うために言葉をさかのぼって作ってる。
……装備は揃ったお。呪文に再挑戦するお

## 第6幕　呪文の解読 ── クライスリ、そして自己関手の圏のモノイド対象

### 6-1　あれは何の合成だったのか ── クライスリ圏

---

**やらない夫**：
まず第3幕の付箋を剥がしに行く。お前はこう言った。
『GetBoss と GetEmail は型が食い違ってて合成できないはずなのに、
FlatMap を挟むと数珠つなぎになる。やる夫たちは何を合成してるんだお』と。
答えを出す準備として、その **つなぎ方そのもの** に名前をつけて関数にしろ。
`A → Option[B]` と `B → Option[C]` を受け取って、
`A → Option[C]` を返す関数だ

**やる夫**：
FlatMap の呼び出しをパターン化するだけだお

```go
// f「してから」g。魚っぽいので釣り演算子と呼びたいお
func Fish[A, B, C any](
    f func(A) Option[B],
    g func(B) Option[C],
) func(A) Option[C] {
    return func(a A) Option[C] {
        return FlatMap(f(a), g)
    }
}

// BossEmail = FindEmployee してから GetBoss してから GetEmail
var BossEmail = Fish(Fish(FindEmployee, GetBoss), GetEmail)
```

Haskellだと `>=>` って演算子で、見た目も魚だお。
`bossEmail = findEmployee >=> getBoss >=> getEmail`

**やらない夫**：
いい。ではここが今日の山場だ。ゆっくりやる。
その Fish を堂々と **合成** と呼びたい。
第5幕で作った基準があるだろう。
『合成がまともにできる世界』の設計基準──圏の公理だ。
Fish が圏の合成の資格を持つための条件を、番号付きで列挙しろ

**やる夫**：
圏の定義と突き合わせながら書くお。
世界の候補はこうだお：対象＝型、射＝ `A → Option[B]` の形の関数ぜんぶ。
ただし矢印としては **A から B への射** とみなす。Option は包装扱いだお。

その上で条件は3つ：
**条件(1)** 射 A→B（実体 A→Option[B]）と射 B→C（実体 B→Option[C]）が
Fish で合成できて、射 A→C（実体 A→Option[C]）になること。
**条件(2)** 合成の **結合法則**：Fish(Fish(f,g),h) = Fish(f,Fish(g,h))。
**条件(3)** 各対象に **恒等射** があること：Fish(id?, f) = f = Fish(f, id?) となる id? の存在

**やらない夫**：
条件(1)は Fish の型シグネチャが既に保証している。
条件(3)から片付けろ。恒等射の候補は？

**やる夫**：
一番素朴な候補は恒等関数 `func(a A) A { return a }` だお。
……ダメだお、型が `A → A` であって `A → Option[A]` じゃない。
この世界の射の形をしてないお。
じゃあ `A → Option[A]` で『何もしない』に一番近いやつ……
値を **ただ包むだけ** の関数だお。`Some` だお！
Writer のときの PureW、Haskellの return と同じ立ち位置のやつだお

**やらない夫**：
では Some が本当に恒等射として働くか、それと条件(2)を、
モナドの3法則──第4幕でお前が『読み飛ばした』と白状したあれ──の隣に並べて書け

**やる夫**：
並べるお。Fish を >=> で書くお。

$$\mathtt{return} \mathbin{>\!\!=\!\!>} f = f, \qquad f \mathbin{>\!\!=\!\!>} \mathtt{return} = f \qquad\Longleftrightarrow\qquad \text{恒等射の単位元則（条件3）}$$

$$(f \mathbin{>\!\!=\!\!>} g) \mathbin{>\!\!=\!\!>} h = f \mathbin{>\!\!=\!\!>} (g \mathbin{>\!\!=\!\!>} h) \qquad\Longleftrightarrow\qquad \text{合成の結合法則（条件2）}$$

…………。

**あーーーー！！！　見えたお！！　全部見えたお！！**

モナドの3法則って、**バラバラの3つのルールじゃなかった** お！
左単位元則と右単位元則は『return がこの世界の恒等射である』こと、
結合法則は『Fish がこの世界の合成である』こと。
3つ合わせて、たった一つのことを言ってるお──

**『失敗するかもしれない関数たちの世界は、圏になる』**

これがモナド則の正体だお！！

**やらない夫**：
その圏には名前がついている。**クライスリ圏** だ。
モナド M ごとに一つ、『A から B への射の実体が A → M B であるような圏』が立つ。
第3幕の付箋の答えも言ってしまえ。お前たちは何を合成していた？

**やる夫**：
**クライスリ圏の射** を合成してたんだお。
`GetBoss` を『Employee から Employee への、失敗込みの矢印』とみなす世界では、
あれは正真正銘、ただの合成だったお。
型が食い違って見えたのは、やる夫が **元の型の圏の目** で見てたからで、
世界の側を掛け替えれば矢印はまっすぐだったお。付箋、回収だお。

それと数えるお。結合法則＋単位元のセット、**これで4回目** だお。
集計のモノイド、Writerのログ、圏の公理、そしてモナド則そのもの。
やらない夫の予告どおり、『あと3回』を数え終わったお

**やらない夫**：
よく数えた。仕上げに実利の答え合わせだ。
第4幕で角括弧ログの Writer もどきが壊した『括り直しリファクタ』。
あれをクライスリ圏の言葉で言い直せ

**やる夫**：
`(find >=> boss) >=> email` を `find >=> (boss >=> email)` に括り直すのは、
クライスリ圏での **合成の結合法則** そのものだお。
角括弧ログ版は結合法則が破れてるから、そもそも圏が立たない。
圏が立たない世界では『途中を関数に括り出す』という
プログラマの基本動作が安全じゃなくなる。
**モナド則を守れ＝お前の作った世界を圏にしろ、って意味だったんだお**

### 6-2　呪文の完全解読 ── 自己関手の圏のモノイド対象

---

**やらない夫**：
ここまでで呪文の容疑者は2人割れている。
『自己関手』＝ Option やスライスの正体、
『モノイド』＝結合法則＋単位元。
残るは『自己関手の **圏** 』と『モノイド **対象** 』の2語だ。一気に落とす。

まず質問。第5幕で圏の定義を作ったとき、対象は何でもいいと言った。
では **対象＝自己関手** という圏は作れるか。
作れるなら、射には何を採用する？

**やる夫**：
対象が Option とかスライスとか Writer で……
その間の矢印は、第5幕で作ったばかりだお。**自然変換** だお。
HeadOption が『スライス → Option』の矢印になる。
合成は自然変換を続けて適用するだけ、恒等射は『何も乗り換えない』変換。
結合法則も単位元則も関数合成から相続する。**圏になるお**。

……こういう『住人を一段上から眺めて、それ自体を点にする』のが
圏論の常套手段っぽいお。モノイドを点1個の圏にしたときと同じ手つきだお

**やらない夫**：
その圏を **自己関手の圏** と呼ぶ。呪文の第3語が落ちた。
次に、この圏には合成とは **別の掛け算** が入っていることに気づいてほしい。
Option とスライス、関手同士を『続けて適用』したら何になる？

**やる夫**：
`[]Option[T]`……Option を食わせたスライス、みたいな **合体関手** ができるお。
関手の合成だお。F∘G も自己関手だから、ちゃんと世界の中に残ってる。
……掛け算って言われると確かに掛け算っぽいお。
結合法則も成り立つお：(F∘G)∘H も F∘(G∘H) も『H して G して F』。
単位元は……何もしない関手、**恒等関手 Id** だお。Id∘F = F∘Id = F。

って、また結合法則＋単位元が出たお！　**自己関手ぜんぶの集まりは、
関手合成を演算とするモノイドみたいになってる** お！

**やらない夫**：
その観察が最後の鍵だ。
いま『モノイドみたい 』と言ったな。第2幕のモノイドは
**集合の中の要素たち** を演算で結合した。
いま起きているのは、**圏の中の対象たち（自己関手）** を ∘ で結合している。
この状況で、第2幕のモノイドに相当するものを圏の中に探したい。
つまり──『自己関手の圏の中で、1個の対象 T が、
掛け算 ∘ に関して **自分ひとりでモノイドの形を再現している** 』とは
どういうことか、を定義したい。
第2幕のモノイドの部品を思い出せ。何と何があった？

**やる夫**：
部品は2つだお。**演算** Combine: 2個のTから1個のT。**単位元** Empty: Tが1個。
これを対象 T が自己関手で、『2個のT』が T∘T になった世界に翻訳すると……

演算にあたるもの：**T∘T から T への矢印**。この圏の矢印は自然変換だから、
自然変換 join: T∘T ⇒ T。
単位元にあたるもの：**単位 Id から T への矢印**、自然変換 pure: Id ⇒ T。

……ちょっと待つお。T∘T → T って、`Option[Option[A]] → Option[A]`……
**第3幕でやる夫が自作した Flatten だお！！**
Id → T は `A → Option[A]`、**Some だお！　return だお！**
演算＝Flatten、単位＝return。とっくに全部持ってたお！

**やらない夫**：
そうだ。あとは法則だけ確認すればいい。
第2幕のモノイドは結合法則と単位元則を要求した。
join と pure に対応する法則を書き下すと、こうなる

$$\text{結合法則：}\quad \mathrm{join} \circ \mathrm{fmap}\,\mathrm{join} \;=\; \mathrm{join} \circ \mathrm{join} \qquad (T{\circ}T{\circ}T \Rightarrow T)$$

$$\text{単位元則：}\quad \mathrm{join} \circ \mathrm{pure} \;=\; \mathrm{id} \;=\; \mathrm{join} \circ \mathrm{fmap}\,\mathrm{pure} \qquad (T \Rightarrow T)$$

結合法則は『三重包装 T∘T∘T を、外側から潰しても内側から潰しても同じ一重に落ちる』。
単位元則は『包んでから潰したら元どおり』。
抽象論で終わらせるな。スライスの三重リストで結合法則を検算しろ。
`xss = [[[1,2],[3]], [[4]]]` だ

**やる夫**：
やるお。三重リスト `[[[1,2],[3]], [[4]]]` を一重まで潰す2ルート。

**内側から**（fmap join：外の各要素の中で先に潰す）：
[[1,2],[3]] → [1,2,3]、[[4]] → [4] で `[[1,2,3], [4]]`、
もう一回 join して **[1,2,3,4]** 。
**外側から**（join：まず外側の連結）：
[[[1,2],[3]], [[4]]] → `[[1,2],[3],[4]]`、
もう一回 join して **[1,2,3,4]** 。

**一致！**　単位元則もやるお。`xs = [1,2]` で、
pure(xs) = [[1,2]] → join → [1,2] ○
fmap pure = [[1],[2]] → join → [1,2] ○
両方とも元どおり。**合格だお**

**やらない夫**：
これで部品が出揃った。組み立てろ。
『圏の中の対象 T が、演算にあたる矢印と単位にあたる矢印を持ち、
結合法則と単位元則を満たすとき、T を **モノイド対象** と呼ぶ』。
ではやる夫、呪文をもう一度、今度はお前が読み上げろ

**やる夫**：
読むお。

**『モナドは自己関手の圏のモノイド対象にすぎない』**

＝ Option やスライスみたいな、Mapと法則を備えた型コンストラクタ（**自己関手**）たちを
対象にして、自然変換を矢印にした世界（**自己関手の圏**）を考える。
その世界には関手合成 ∘ という掛け算がある。
ある自己関手 T が、Flatten（join: T∘T ⇒ T）と return（pure: Id ⇒ T）を持ち、
それが結合法則と単位元則を満たすなら、
T は第2幕の Monoid インターフェースの **一段上の世界版** を実装している（**モノイド対象**）。
その T を **モナド** と呼ぶ。

──ただの日本語だお。呪文じゃなくて、**定義を圧縮した略語** だったお。
しかも中身は全部、やる夫が自分の手で作った部品だお。
Flatten は第3幕、モノイドは第2幕、関手と自然変換は第5幕。
悔しいけど『何か問題でも？』って言いたくなる気持ち、今なら分かるお

**やらない夫**：
原典も見せておこう。マクレーンの教科書の一節はこうだ。

*All told, a monad in X is just a monoid in the category of endofunctors of X,
with product × replaced by composition of endofunctors and
unit set by the identity endofunctor.*[^2]

後半までちゃんと読めば『積 × を関手合成に、単位を恒等関手に置き換えて』と
種明かしまで書いてある。ジョークで切り取られたのは前半だけだ

**やる夫**：
原文は親切だったのかお……。切り抜きはいつの時代も罪だお。
ところで bind（>>=）と join、どっちが本物のモナドなんだお。
Haskellの class Monad は >>= で、数学は join らしいけど

**やらない夫**：
相互に変換できるから等価だ。
`m >>= f = join (fmap f m)`、逆に `join mm = mm >>= id`。
数学側は join を好み（法則が図式で書きやすい）、
プログラマは >>= を好む（パイプラインが書きやすい）。同じ山の別ルートだ。

歴史も一段落で押さえておけ。
この概念は1950年代末に代数トポロジーの計算道具として生まれ
（ゴドマンの『標準構成』）、後にモナドという名で定着した[^11]。
1989〜91年、モッジがこれを **プログラムの意味論** ──
『例外・状態・非決定性などの計算効果を統一的に記述する枠組み』として再発見し[^12]、
ワドラーが1992年ごろ、それを **Haskellのプログラミング技法** に翻訳した[^13]。
数学者が30年温めた抽象が、お前の nil 地獄とログ地獄を救う道具として着地するまで、
だいたい一世代かかっている

**やる夫**：
純粋数学 → 意味論 → 実務ツールで30年かお。
やる夫がバッチの障害対応で再発明した道が、歴史の再放送だったんだお

### 7-1　型の代数 ── 積・余積・終対象・始対象

---

**やらない夫**：
最初は型そのものの作り方だ。基本の質問から。
型 A と B があるとき、『A と B を **両方** 持つ型』を作る一番まっとうな方法は？

**やる夫**：
Goなら struct、Haskellならタプル (A, B) だお。
`struct{ Name string; Age int }` みたいに、フィールドを並べるだけだお

**やらない夫**：
圏論では A×B と書いて **積** と呼ぶ。だが定義の流儀が独特だ。
『フィールドを2つメモリに並べたもの』とは **言わない**。こう言う。

『2本の射影 `fst: A×B → A` と `snd: A×B → B` を備え、
同じ2本を持つどんな型からも、ちょうど1本の射で A×B に辿り着ける型』。

中身の実装ではなく、**外に出ていく矢印の仕様** で型を特徴づける。
この流儀を **普遍性** と呼ぶ。第5幕の『対象の中身は見ない』の実践編だ

**やる夫**：
インターフェースだけ定義して『これを満たす奴が積だ』って言ってる感じだお。
実装が struct だろうとポインタのペアだろうと、
fst と snd が取り出せて一意なら積。プログラマには馴染みのある発想だお

**やらない夫**：
では逆を作れ。積の矢印を全部逆向きにする。
射影 `A×B → A` を注入 `A → A+B` にひっくり返した型は何だ

**やる夫**：
入る向きが逆……『A **または** B』かお？
`A` からも `B` からも作れて、でき上がったものは
中身がどっちか一方。Haskellなら `Either A B` だお

```haskell
data Either a b = Left a | Right b   -- A + B（余積）

-- 使う側は「どっちか」を必ず場合分けさせられる
describe :: Either Int String -> String
describe (Left  n) = "数値 " ++ show n
describe (Right s) = "文字列 " ++ s
```

**やらない夫**：
それが **余積**（直和、コプロダクト）だ。A+B と書く。
ここで言語比較の急所を押さえておけ。Goで『A または B』を書いてみろ

**やる夫**：
Goには直和型が **ない** お。だから……
`(value T, err error)` の2値返しとか、
`interface{}` に入れて型switch、とかで **運用でカバー** してるお

```go
// Goの「余積もどき」。両方 nil / 両方非 nil のバグを型で防げない
type Result struct {
    Value *int
    Err   *string
}
```

両方 nil や両方セットの事故を、型が止めてくれないお。何度も踏んだお

**やらない夫**：
そこが本質だ。**積（struct）はどの言語にもあるが、余積（直和）は無い言語が多い**。
識者が『**代数的データ型**（ADT）』と言ったら、
積と余積を自由に組み合わせて型を組み立てられる仕組みのことだ。
第3幕の Option も第4幕の Writer も、正体は積と余積の組み合わせだぞ

**やる夫**：
言われてみれば Option[T] は『T（値あり）**または** 空』で余積、
Writer は『値 **かつ** ログ』で積だお。
やる夫が作った器は、全部この2部品でできてたんだお

**やらない夫**：
そのとおり。ついでに極端な2つを拾って幕を締める。
フィールドが **0個** の struct、Goの `struct{}` は何だ

**やる夫**：
値が1個しかない型だお。中身の情報量ゼロ。HaskellだとUnit、`()` だお。
Goでは `map[string]struct{}` で集合を作るときに使うお

**やらない夫**：
圏論では **終対象** と呼ぶ。
どの型からも『全部 `()` に潰す関数』がちょうど1本入ってくる、矢印の終着駅だ。
双対に、値が **0個** の型──Haskellの `Void`──は **始対象**。
どの型へもちょうど1本の射が出ていく矢印の始発駅で、
『起きえない値』『到達不能コード』の型付けに使う。

整理する。**積・余積・終対象・始対象**、この4つは
第5幕の『矢印の仕様で対象を決める』普遍性の、一番易しい実例だ。
型を『実装』でなく『満たすべき仕様』で見る目が、これで手に入った

**やる夫**：
データモデリングを圏の言葉で見直せた気分だお。次は何だお

### 7-2　二本の階段 ── カインドと型クラス階層

---

**やらない夫**：
道具が増えてきたから、ここで棚に格付けの秩序を入れる。
識者が『それは **Semigroup で十分**』とか『**Kind が合ってない**』と言うだろう。
あの格付けの正体だ。地図には **二本の階段** がある。お前はもう両方登っている

**やる夫**：
二本……値の世界と、文脈の世界かお？

**やらない夫**：
そうだ。まず **値の階段**（第2幕）。

**半群 ⊂ モノイド ⊂ 群**。

・半群（Semigroup）：結合法則を満たす演算 `<>` だけ。単位元も逆元も要らない
・モノイド：それに単位元 `mempty` を足したもの
・群：さらに逆元を足したもの

『畳むバッチが絶対に空にならない』と保証できるなら、単位元は要らない。
だから識者の『Semigroup で十分』は、**要求を1段ゆるめる倹約** の話だ

**やる夫**：
第2幕の判定表で、maxが『単位元だけ無い惜しい奴』だったお。
あれはモノイドにはなれないけど **半群ではある**、って格付けだったんだお

**やらない夫**：
正確だ。もう一本、**文脈の階段**（第5〜6幕）。

**関手 ⊂ アプリカティブ ⊂ モナド**。

・関手：`fmap` で中身を変換できるだけ
・アプリカティブ：独立した複数の文脈を束ねられる（次節でやる）
・モナド：前の結果を見て次を分岐できる

**上に行くほど強く、下に行くほど広い**。上の階級は下の能力を全部含む

**やる夫**：
強いほど使える型が減って、弱いほど当てはまる型が増えるお。
インターフェースのメソッドが多いほど実装が減るのと同じ、第5幕の理屈だお。
……で、『Kind が合ってない』の **カインド** って何だお

**やらない夫**：
**型の型** だ。第4幕でお前が『作りかけの型』と呼んだものの、正式な戸籍だ。

・`Int` や `String` のような完成した型のカインドは `*`
・`Maybe` や `[]` のような『型を1個食べて型を返す』もののカインドは `* -> *`
・`Either` や `Map` のような『2個食べる』もののカインドは `* -> * -> *`

`Functor` のインスタンスになれるのは、カインドが `* -> *` の型だけだ。
`Int`（カインド `*`）を Functor にしようとすると『Kind が合わない』と叱られる

```haskell
class Functor (f :: * -> *) where   -- f は「型を1個食べる」ものに限る
  fmap :: (a -> b) -> f a -> f b

instance Functor Maybe where ...    -- Maybe :: * -> * なのでOK
-- instance Functor Int where ...   -- Int :: * なので Kind エラー
```

**やる夫**：
値の型検査が一段上でも走ってるお。
`fmap` に `Int` を渡そうとするのは、
`func(int) int` に文字列を渡すのと同じ種類の間違いってことだお。階が違うだけだお

**やらない夫**：
その通り。最後に、実務で毎日効く小道具 **newtype** を渡す。
第2幕の判定表の後始末だ。`Int` は `+` でも `×` でもモノイドだったな。
だが1つの型に2つのモノイド実装は登録できない。どっちの `<>` か曖昧になる。
そこで薄い包み紙で **同じ中身を別のモノイドとして** 包み分ける

```haskell
newtype Sum     = Sum     Int   -- <> は +,  mempty は 0
newtype Product = Product Int   -- <> は ×,  mempty は 1
newtype Any     = Any     Bool  -- <> は ||, mempty は False
newtype All     = All     Bool  -- <> は &&, mempty は True

foldMap Sum     [1,2,3,4]  -- Sum 10   （合計）
foldMap Product [1,2,3,4]  -- Product 24（総積）
foldMap All     [True,False,True] -- All False（全部Trueか？）
```

**やる夫**：
`foldMap` に渡す包み紙を変えるだけで、合計にも総積にも全称判定にもなるお。
第2幕で作った判定表の各行に、正式な住所と名札がついた感じだお。
`Any`/`All`/`Min`/`Max`/`First`/`Last` ──『すごいHaskell』のモノイド章の顔ぶれは、
ぜんぶこの包み分けの実例だったんだお。土台が固まったお

### 7-3　独立を束ねる ── アプリカティブ

---

**やらない夫**：
文脈の階段の真ん中、アプリカティブに降りて歩く。
第2の階段で『関手とモナドの中間』と言ったが、抽象論では身につかない。
題材はお前の得意な入力検証だ。ユーザー登録フォームの3項目を検証する

**やる夫**：
名前・年齢・メール、それぞれ検証して、全部OKなら User を作るお。
モナドの `>>=` でやるお

```haskell
mkUser <$> validateName n >>= \nm ->
       validateAge  a >>= \ag -> ...  -- 直列でつなぐ
```

**やらない夫**：
そのモナド版には欠陥がある。`>>=` は
**前の結果が出てから次を始める** 直列合成だ。
名前がエラーで落ちたら、年齢とメールの検証は **走りもしない**。
フォームで『まず名前を直せ、直したら次は年齢だ』と1個ずつ叱られたら、お前どう思う

**やる夫**：
最悪だお。3個まとめてエラー表示してほしいお。
……でも `>>=` は前を見てから次だから、最初のエラーで打ち切られるお。
3項目は本当は **お互い独立** なのに、直列の道具を使ったせいで打ち切られてるお

**やらない夫**：
そこでアプリカティブだ。核心は演算子2つ

```haskell
(<$>) :: (a -> b)   -> f a -> f b   -- fmap の別名（第5幕の Map）
(<*>) :: f (a -> b) -> f a -> f b   -- 「文脈に入った関数」を「文脈に入った値」に適用
```

`<*>` が新顔だ。**関数の側まで文脈に入っている** のがミソだ。
3引数関数 `mkUser :: Name -> Age -> Email -> User` に、検証済みの各値を順に食わせる

```haskell
mkUser <$> validateName n <*> validateAge a <*> validateEmail e
    :: Validation [Error] User
```

**やる夫**：
3つの `validate` が **お互いを待ってない** お。並べて全部走らせて、
エラーは全部集めてから、成功が揃ったときだけ User を組む。
具体的に入れてみるお。名前OK・年齢NG・メールNGなら──

```haskell
-- Validation は「成功なら値、失敗ならエラーを << でモノイド結合」
Success mkUser <*> Success 30日目...
-- 実際：
mkUser <$> Success "yaruo" <*> Failure ["年齢が負"] <*> Failure ["@が無い"]
    == Failure ["年齢が負", "@が無い"]   -- エラー2件が合流！
```

エラーが `["年齢が負", "@が無い"]` と **モノイドで合流** したお！
……あっ。ここでも第2幕のモノイドが効いてるお。
エラーを溜める器がモノイド（リストの連結）だから、失敗を全部足し合わせられるんだお

**やらない夫**：
鋭い。**アプリカティブの失敗蓄積は、エラー型がモノイドであることに乗っている**。
もう一つ顔を見せる。リストをアプリカティブとして使うと **直積** が出る

```haskell
(,) <$> [1,2] <*> ['a','b']
    -- [(1,'a'), (1,'b'), (2,'a'), (2,'b')]
```

for の二重ループが1行だ。SQLのクロス結合と同じ形。
『独立した選択肢の全組み合わせ』はアプリカティブの十八番だ。
この『独立か、依存か』の線引きが、第2の階段の中段の意味だ[^21]

**やる夫**：
Functor と Monad の中間、って抽象的な位置づけかと思ってたら、
『各ステップが独立なら中段（並列・全集め）、前を見るなら上段（直列・打ち切り）』
っていう **すごく実務的な装備選択** だったんだお

### 7-4　構造をひっくり返す ── Foldable と Traversable

---

**やらない夫**：
アプリカティブを手に入れた今だからできる技をやる。
実務で一番『あってくれ』と唱える操作だ。お前も踏んだことがあるはずだ。
`[Maybe a]` を手にして、`Maybe [a]` が欲しくなったことは

**やる夫**：
あるお！　IDのリストを全部引いて、
**全部成功なら社員リスト、1個でも失敗なら全体を失敗** にしたいお。
でも手元にあるのは『成功したりしなかったりの結果のリスト』`[Maybe Employee]` で、
欲しいのは `Maybe [Employee]`。**層が内と外で逆さま** だお

**やらない夫**：
その『ひっくり返し』が **traverse**、特殊形が **sequenceA** だ

```haskell
sequenceA [Just 1, Just 2, Just 3]   -- Just [1,2,3]
sequenceA [Just 1, Nothing, Just 3]  -- Nothing（1個でもダメなら全体ダメ）

traverse validateUser users          -- Maybe [ValidUser]
traverse (\x -> if x>0 then Just x else Nothing) [1,2,3]  -- Just [1,2,3]
traverse (\x -> if x>0 then Just x else Nothing) [1,-2,3] -- Nothing
```

**やる夫**：
`sequenceA` の中身が知りたいお。どうやってひっくり返してるんだお

**やらない夫**：
まさに前節のアプリカティブだ。リストを1個ずつ、`<*>` で束ねていく

```haskell
sequenceA []     = pure []
sequenceA (x:xs) = (:) <$> x <*> sequenceA xs
    -- 「先頭の Maybe」と「残りを畳んだ Maybe」を <*> で合体
```

`traverse f = sequenceA . map f`。
だから traverse が要求するのは『中身を束ねる器がアプリカティブであること』。
`Maybe` でも `Either` でも `Validation` でも `IO` でも動く。
`traverse print xs` なら『各要素を印字しつつ、結果を1つの IO にまとめる』だ

**やる夫**：
1個の道具で、検証も、印字も、DBアクセスの一括実行もいけるお。
……traverse の『層を入れ替える』って、第6幕の join に似てるお。
join は `M(M a) → M a` で **同じ層を潰す**、
traverse は `[M a] → M [a]` で **違う層を入れ替える**。親戚だお

**やらない夫**：
良い観察だ。ついでに片割れの **Foldable** も回収しておけ。
『畳んでモノイドに潰せる』構造がFoldableだ。第2幕の `FoldMapPar` の正体でもある

```haskell
foldMap Sum [1,2,3]  -- Sum 6
sum   [1,2,3]        -- 6      （Foldable の便利関数）
toList tree          -- 木を畳んでリストに
length, elem, maximum ... -- 全部 Foldable の能力
```

**やる夫**：
Foldable は第2幕の畳み込みの、Traversable はアプリカティブの応用だお。
『すごいHaskell』の Traversable の章、開かずに投げたけど、
やりたいことは『リストの Maybe を Maybe のリストにする』だけの身近な話だったのかお

### 7-5　糖衣をはがす ── do記法

---

**やる夫**：
ここで長年の疑問を潰したいお。Haskellの **do記法**。
あれのせいで『Haskellは実は手続き型言語』みたいな混乱記事を量産させられたお。
正体は何だお

**やらない夫**：
`>>=` にかぶせた砂糖だ。それ以上でも以下でもない。左が do、右が素の姿

```haskell
do x <- mx              mx >>= \x ->
   y <- my        ==     my >>= \y ->
   return (x + y)        return (x + y)
```

矢印 `<-` は『文脈から中身を取り出して名前をつける』。
脱糖すると全部 `>>=` のラムダのネストに戻る。手続きに見えて、中身は関数だ

**やる夫**：
じゃあ Maybe で書いた BossEmail を do にすると……
第3幕の FlatMap 地獄が、こう見えるお

```haskell
bossEmail id = do
  e <- findEmployee id   -- どれか一つでも Nothing なら
  b <- getBoss e         -- 全体が Nothing に短絡する
  getEmail b
```

`if nil return nil` が **1個も見えない** お。
短絡は `>>=` の中に1回だけ書いてあって、使う側からは消えてるお。第3幕の到達点だお

**やらない夫**：
そうだ。もう一つ、リストモナドで書くと **内包表記** と同じものになるのを見せる

```haskell
do x <- [1,2,3]              [ (x,y) | x <- [1,2,3]
   y <- [10,20]        ==              , y <- [10,20]
   guard (x /= 2)                      , x /= 2 ]
   return (x, y)
   -- [(1,10),(1,20),(3,10),(3,20)]
```

**やる夫**：
`guard` は前節の直積を **条件で刈り込む** やつだお。
do の見た目が手続き的なだけで、実体は第6幕の `>>=` の数珠つなぎ、
つまり **クライスリ圏の射の合成** そのものなんだお。
『Haskellは手続き型』って記事に、もう騙されないお

### 7-6　状態を持ち運ぶ ── State と関数三兄弟

---

**やらない夫**：
第4幕で Reader と Writer に会ったな。三男を紹介する。**State** だ。
三兄弟の共通点は、全員 **中身が関数のモナド** ──『箱』アナロジーが死ぬ一族だ

```haskell
newtype Reader r a = Reader { runReader :: r -> a }        -- 環境を読む（第4幕）
newtype Writer w a = Writer { runWriter :: (a, w) }        -- ログを書く（第4幕、wはモノイド）
newtype State  s a = State  { runState  :: s -> (a, s) }   -- 状態を読んで書く
```

**やる夫**：
先に痛みを見せてほしいお。State が無いと何が辛いんだお

**やらない夫**：
Goでスタック電卓を書け。状態（スタック）を手で引き回してみろ

**やる夫**：
戻り値のたびにスタックを一緒に返して、次に手渡して……

```go
func push(stack []int, x int) []int           { return append(stack, x) }
func pop(stack []int) (int, []int)            { return stack[len(stack)-1], stack[:len(stack)-1] }

s := []int{}
s = push(s, 3)
s = push(s, 5)
a, s := pop(s)
b, s := pop(s)       // s の引き回しを1回でも書き忘れたら壊れる
s = push(s, a+b)
```

`s = ...; s = ...` の **バケツリレー** が全行に染み出すお。
1回でも受け取り忘れたら古いスタックを使って事故るお。先月やったお

**やらない夫**：
State モナドは、そのバケツリレーを `>>=` の裏に隠す。同じ電卓がこうなる

```haskell
pop  :: State [Int] Int
push :: Int -> State [Int] ()

calc :: State [Int] ()
calc = do
  push 3
  push 5
  a <- pop
  b <- pop
  push (a + b)      -- 状態の受け渡しは do が自動でやる

runState calc []    -- ((), [8])
```

**やる夫**：
`s =` が **全部消えた** お！　状態は見えないまま正しく流れてるお。
乱数の seed、パーサの残り入力、ゲームのスコア……
『手で引き回すと事故る隠し引数』はみんな State に載るんだお。
第4幕で `Reader[int]` の中身が未来にしか無くて面食らったけど、
三兄弟の顔つき（`環境→値`、`値と付録`、`状態→値と状態`）が分かれば、もう怖くないお

**やらない夫**：
三兄弟は次幕のトランスフォーマーで **合体** させることになる。顔を覚えておけ

### 7-7　行って戻れる最良ペア ── 随伴とカリー化

---

**やらない夫**：
教養装備の最後は、識者が口にする一番怖い単語 **随伴**（アジョイント）だ。
だが最頻出の実例を、お前はほぼ確実に書いたことがある。
2引数関数の引数を1個ずつ渡せるようにする書き換えを何と呼ぶ

**やる夫**：
カリー化だお。Goでも書けるお

```go
func Curry[A, B, C any](f func(A, B) C) func(A) func(B) C {
    return func(a A) func(B) C {
        return func(b B) C { return f(a, b) }
    }
}
// 逆向きも書ける
func Uncurry[A, B, C any](f func(A) func(B) C) func(A, B) C {
    return func(a A, b B) C { return f(a)(b) }
}
```

**やらない夫**：
Curry と Uncurry は **完全に元通りに行き来できる**（`Uncurry . Curry = id`）。
これが主張している型の対応を書くと、こうだ

$$\mathrm{Hom}(A \times B,\; C) \;\cong\; \mathrm{Hom}(A,\; B \to C)$$

左辺『A×B から C への関数』の集合と、
右辺『A から〈B→C〉への関数』の集合が、**1対1に、かつ自然に対応する**。
このように2つの構成──ここでは『**B を掛ける**（積）』と『**B→ の関数型を作る**（冪）』──が
Hom集合の1対1対応で結ばれる関係を **随伴** と呼ぶ[^14]

**やる夫**：
第7-1でやった **積** が、随伴の片側に顔を出してるお。
道具がつながってきたお

**やらない夫**：
随伴は深入りしない。今幕の獲得目標は一行でいい。
**随伴＝カリー化のような『行って戻れる最良のペア』の一般化**。
識者が『それ随伴ですね』と言ったら、
『2つの操作が過不足なく対応してる、という指摘だな』と読めれば十分だ

**やる夫**：
怖い単語の実例が毎日書いてるカリー化なら、急に親しめるお。
……教養装備、7つ揃ったお。
型の代数、二本の階段、アプリカティブ、Traversable、do、State、随伴。
必須5個の隣が、だいぶ賑やかになったお

**やらない夫**：
ここまでが『**降りて歩く**』道具、日々の実務で本当に使う教養装備だ。
次幕は毛色が変わる。**降りずに車窓から眺める** 高等呪文──
勉強会で識者が振り回すが、正体さえ分かれば怖くないバズワード群だ。
名所ごとに写真は撮る。つまり具体例は見る。だが再発明のサイクルはもう回さない

望むところだお。勉強会で識者が振り回す難語、まとめて顔を覚えたいお。
モナドトランスフォーマー、フリーモナド、レンズ、コモナド……。
『すごいHaskell』[^15]も後半で力尽きて、Zipperの章は開いてすらいないお

初見の怪物に見えても、正体は旧知の構造だ。それを毎回確認していく。

### 8-1　モナドを積む ── モナドトランスフォーマー

---

**やる夫**：
最初が界隈の本丸 **モナドトランスフォーマー** かお。名前が強すぎて避けてたお

**やらない夫**：
動機は切実だ。前幕で三兄弟に会った。実務のハンドラは、
状態（State：セッション）も、失敗（Except：エラー）も、IO（DB叩き）も、**全部同時に** 欲しい。
だが1つのモナドに全能力は入っていない。まず、道具なしで束ねる痛みを見ろ。
『状態を持ちつつ失敗しうる計算』を手で書くとこうなる

```haskell
-- State と Either を手で編む：状態を通しつつ、失敗も表現する
type Handler s e a = s -> Either e (a, s)

andThen :: Handler s e a -> (a -> Handler s e b) -> Handler s e b
andThen h f = \s -> case h s of
  Left err      -> Left err            -- 失敗ならそこで打ち切り
  Right (a, s') -> f a s'              -- 成功なら状態を次へ手渡す
```

**やる夫**：
うわ、`case` で失敗を捌きつつ、成功したら状態 `s'` を手で次に渡してるお。
これ、State の面倒（状態リレー）と Either の面倒（失敗分岐）を **両方同時に手書き** してるお。
効果が3つ4つになったら、この `case` のネストで死ぬお

**やらない夫**：
その手書きを部品化したのがトランスフォーマーだ。
『土台のモナド `m` を1段くぐって、能力を1つ足す』専用部品。末尾に T がつく

```haskell
StateT  s m a   -- m の上に「状態」を積む   （中身は s -> m (a, s)）
ExceptT e m a   -- m の上に「失敗」を積む   （中身は m (Either e a)）
ReaderT r m a   -- m の上に「環境」を積む   （中身は r -> m a）

-- 3階建ての塔：状態 + 失敗 + IO を1本のモナドに
type App a = StateT Session (ExceptT Error IO) a

login :: Credentials -> App User
login c = do
  sess <- get                      -- StateT の能力（状態）
  when (bad c) $ throwError AuthErr -- ExceptT の能力（失敗）
  liftIO $ putStrLn "logged in"     -- IO の能力（一番下の階）
  pure (userOf c)
```

**やる夫**：
`get` も `throwError` も `liftIO` も **同じ do の中で** 使えてるお。
さっき手書きした `case` の山が、塔を積むだけで消えたお

**やらない夫**：
ただし塔には2つの罠がある。実務で刺さるやつだ。
罠1：下の階の能力を呼ぶには本来 `lift` で潜る必要がある。
`liftIO` はその一種だ。階が深いと `lift . lift . lift` の梯子地獄になる。
これを隠すのが **MTL** という型クラス群（`MonadState`, `MonadError`, `MonadIO`）だ[^16]。
『どの階の能力か』を型クラスが自動解決して、`get` や `throwError` を素で書けるようにする。
識者の『MTLスタイル』はこれだ

**やる夫**：
梯子を型クラスが登ってくれるのかお。じゃあ罠2は

**やらない夫**：
**積む順番で意味が変わる**。これが本当に刺さる。
`StateT s (ExceptT e IO)` と `ExceptT e (StateT s IO)`、
失敗したときに **状態が巻き戻るかどうか** が逆になる

```haskell
-- StateT s (ExceptT e IO)  : 失敗すると (a,s) ごと Left に消える → 状態も巻き戻る
-- ExceptT e (StateT s IO)  : 状態は外側に残る          → 失敗しても状態は生き残る
```

**やる夫**：
トランザクションで例外時に巻き戻したいか、ログだけは残したいかで、
**塔の積み順を選ぶ** ってことかお。順番がセマンティクスなんだお。難しいわけだお

**やらない夫**：
そして根っこは第6幕だ。**モナドは一般に合成できない**
──`Option` と `State` を素朴に重ねても圏になる保証がない。
だから『1段ずつ手で積む部品』が要る。トランスフォーマーは、
第6幕の『モナドは素直に合成できない』という含みが、現実の設計問題として返ってきた姿だ

**やる夫**：
呪文を解いたときの含みが、こんな所で牙を剥くとはお。旧知の構造だお

### 8-2　プログラムをデータにする ── フリーモナド

---

**やる夫**：
次は **フリーモナド**。『プログラムをデータとして組み立てて後から実行』
って聞いたけど、意味が分からんお

**やらない夫**：
痛みから入る。前節のトランスフォーマーで `IO` を土台に置くと、
`login` は **実行と記述がべったり癒着** する。テストで本物のDBを叩きたくないだろう。
そこで発想を変える。**やりたい命令を、実行せず『データ』として書き並べる**。
まず命令を1個の関手として定義する

```haskell
-- コンソール操作の「命令」。next は「この命令の後に続く残りのプログラム」
data ConsoleF next
  = PutLine String next          -- 出力して続ける
  | GetLine (String -> next)     -- 入力を受けて続ける
  deriving Functor

type Console a = Free ConsoleF a  -- ConsoleF 命令を数珠つなぎにしたデータ
```

`Free f` は雑に言えばこういう再帰データだ

```haskell
data Free f a = Pure a               -- 「もう命令は無い、結果は a」
              | Free (f (Free f a))  -- 「命令 f、その中に続きが入っている」
```

**やる夫**：
`Pure` が終端で、`Free` が『命令＋続き』の入れ子……
これ、**命令を要素にした連結リスト（というか木）** だお。
`program` を書いても、走らずに **この木が組み上がるだけ** ってことかお

**やらない夫**：
そのとおり。do記法で書けるから見た目は普通のプログラムだが、中身はデータだ

```haskell
program :: Console ()
program = do
  putLine "名前は?"
  n <- getLine
  putLine ("こんにちは " ++ n)
-- これは「実行結果」ではなく、PutLine → GetLine → PutLine という命令の木そのもの
```

木ができただけでは何も起きない。**インタプリタ** を後から付ける。
同じ `program` に、本番用と、テスト用と、2つの意味を与える

```haskell
-- 本番：実際に端末とやりとりする
runIO :: Console a -> IO a
runIO (Pure a)                 = pure a
runIO (Free (PutLine s next))  = putStrLn s >> runIO next
runIO (Free (GetLine k))       = getLine   >>= runIO . k

-- テスト：入力を用意し、出力を溜める。IOを一切使わない
runPure :: [String] -> Console a -> (a, [String])   -- (結果, 出力ログ)
runPure ins       (Pure a)                = (a, [])
runPure ins       (Free (PutLine s next)) = let (a, out) = runPure ins next in (a, s:out)
runPure (i:ins)   (Free (GetLine k))      = runPure ins (k i)
```

**やる夫**：
`runPure ["yaruo"] program` を手で追うお。
PutLine \"名前は?\" → GetLine に \"yaruo\" を渡す → PutLine \"こんにちは yaruo\"。
結果は `((), [\"名前は?\", \"こんにちは yaruo\"])` だお。
**IOを一度も使わずに、出力を値として検査できた** お。テストが書けるお！

**やらない夫**：
それがFreeの眼目だ。**記述（program）と解釈（run*）の分離**。
同じ記述をIOでもテストでも走らせられるし、
命令の木を **実行前に覗いて最適化する** ことすらできる（データだからな）。
そして第6幕とつながる。`Free f` は
**どんな命令関手 f にも、モナド則を満たす最小のモナドを機械的にかぶせる** 構成だ。
お前が第6幕で手作業で確認したモノイド対象の法則を、Freeは自動で持たせてくれる。
定番の出典は Swierstra の *Data types à la carte* だ[^17]

**やる夫**：
第6幕で手で組んだモナドを、命令を放り込むだけで自動生成してくれるのかお。
……あっ、でもこの `runPure`、命令が左に深くネストすると
毎回頭から辿り直しになって遅そうだお

**やらない夫**：
よく気づいた。素朴なFreeは左結合の `>>=` で二次関数的に遅くなる欠点がある。
その改良版が、後で出てくる **代数的効果** につながる。伏線として置いておけ

### 8-3　効果を型クラスで書く ── Tagless Final

---

**やらない夫**：
8-2のFreeは、命令をデータにしたから覗けた。だが構造を覗く必要がなく、
本番実装とテスト実装を差し替えたいだけなら、木を作るコストは払いすぎだ。
Goで同じ分離をするなら、コードを見ずに骨格を書け

**やる夫**：
日常業務だお。能力をinterfaceにして、処理は具体型じゃなくinterfaceを受け取るお

```go
type Console interface {
    PutLine(s string)
    GetLine() string
}
func program(c Console) {
    c.PutLine("名前は?")
    n := c.GetLine()
    c.PutLine("こんにちは " + n)
}
// 本番は端末実装、テストはモック実装を渡す
```

**やらない夫**：
そのGoの設計を、型クラスと多相なプログラムへ翻訳したものが **Tagless Final** だ。
Freeの `ConsoleF` データ型の代わりに、能力をインターフェースとして宣言する

```haskell
-- 「コンソール能力を持つモナド m」というインターフェース
class Monad m => Console m where
  putLine :: String -> m ()
  getLine :: m String

-- program は「Console 能力を持つ何か m」に対して多相
program :: Console m => m ()
program = do
  putLine "名前は?"
  n <- getLine
  putLine ("こんにちは " ++ n)
```

解釈は **インスタンス** で与える。本番は IO、テストは State

```haskell
instance Console IO where
  putLine = putStrLn
  getLine = Prelude.getLine

instance Console (State ([String],[String])) where   -- (入力, 出力ログ)
  putLine s = modify (\(i,o) -> (i, o ++ [s]))
  getLine   = state (\(i:is, o) -> (i, (is, o)))
```

**やる夫**：
……これ **Goのインターフェースそのもの** だお。
`Console` がインターフェース、`instance` が実装、`program` は
『Console を満たす何か `m` に依存する処理』。
本番実装とモック実装を差し替える、やる夫が毎日やってる **依存性注入（DI）** だお！

**やらない夫**：
予想と骨格が完全に一致したな。違うのは、Goでは値として `Console` を渡し、
こちらでは『`Console` 能力を持つ `m`』という型制約で渡す点だ

**やる夫**：
じゃあ Free と Tagless、どっちを使えばいいんだお。同じことができるお

**やらない夫**：
トレードオフだ。
・**Free**：プログラムが **データ** だから、実行前に覗ける・変形できる・最適化できる。
代償は、その内省を使わないなら overhead が大きく、素朴実装は遅い。
・**Tagless Final**：プログラムが **関数** だから速く素直。だが実行前に構造を覗けない。

第4幕でお前がやった interface 差し替えのモックは、実は Tagless Final の素朴版だ。
名前がついて厳めしく見えるだけで、お前は何年も前からこれを書いていた

**やる夫**：
効果をデータで持つか関数で持つか、の二択だったんだお。
Goしか知らなくても、片方（DI）はもう体で分かってたお

### 8-4　トランスフォーマーの次世代 ── 代数的効果

---

**やらない夫**：
効果4連星の締めだ。8-1のトランスフォーマーの罠──
塔の積み順がセマンティクスを縛り、深いと梯子地獄──を覚えているな。
その現代的な回答が **代数的効果（algebraic effects）** だ。
効果を第一級にして、**ハンドラ** で意味を与える。まず使う側

```haskell
-- 効果を「宣言」だけする（どう動くかは書かない）
program :: (State Int :< es, Except String :< es) => Eff es Int
program = do
  n <- get
  when (n < 0) $ throwError "負だ"
  put (n * 2)
  pure n
```

`program` は『State と Except の効果を使う』とだけ言い、動作は未定義だ。
動作は **ハンドラ** が外から与える。しかも **積み順に縛られず** 差し替えられる

```haskell
runProgram :: Int -> Either String (Int, Int)
runProgram s0 = run . runError . runState s0 $ program
--                     ^^^^^^^^   ^^^^^^^^^^^  ← ハンドラの順を選べる
-- runState を外にすれば失敗時に状態が残り、内にすれば巻き戻る、を後から選択できる
```

**やる夫**：
8-1では **塔を組む時点** で巻き戻すか決め打ちだったのが、
代数的効果だと **ハンドラを掛ける順** で後から選べるお。柔らかくなってるお

**やらない夫**：
そして8-2のFreeとつながる。
代数的効果の実装の芯は、Freeを効率化・一般化した **freer** という構造だ。
『効果を投げて、外側のハンドラが捕まえて意味を与える』のは、
例外の throw/catch を、例外以外のあらゆる効果に広げたものだと思え。
言語・ライブラリはこのあたり──
Koka、OCaml 5 のエフェクトハンドラ、Unison、Haskellの polysemy・effectful。
理論の出発点は Plotkin と Pretnar だ[^20]

**やる夫**：
トランスフォーマーの塔とFreeの木、その合流点で
いま人類が工事してる現場ってことかお。名所というより **建設中の橋** だお

**やらない夫**：
そのとおり。ここは第7幕で引く国境線の、**内側ギリギリ** だ。
実務で今すぐ使い倒す道具というより、動向を知っておく最前線だな

### 8-5　ネストの奥を狙い撃つ ── レンズ / Optics

---

**やる夫**：
効果の話から離れて、次は **レンズ**。名前から中身が全く想像つかないお

**やらない夫**：
痛みから入る。不変データで、深いネストの1フィールドだけ更新しろ。
`会社 → 部署 → マネージャ → メール` を書き換える。まずGoで

**やる夫**：
不変だから、内側を作り直しては外側を包み直して……

```go
// 変えたい1フィールドのために、通り道の構造体を全部作り直す
newCompany := company
newDept := company.Dept
newMgr := company.Dept.Manager
newMgr.Email = "new@x.com"
newDept.Manager = newMgr
newCompany.Dept = newDept
```

変えない部分まで **全部手で写経** するお。3階でこれだから、5階なら発狂だお。
Haskellのレコード更新構文でも `company{ dept = ... }` のネストで同じ地獄だお

**やらない夫**：
その痛みを消すのがレンズだ。1フィールドへの
**getter と setter を1個の合成可能な部品に束ねたもの**。まず使い心地

```haskell
view  (dept . manager . email) company            -- 取り出す（getter）: "old@x.com"
set   (dept . manager . email) "new@x.com" company -- 差し替え、新しい company を返す
over  (dept . manager . email) toLower company     -- 関数で更新
```

**やる夫**：
**レンズを `.` で合成すると奥に潜れる** お！
`dept . manager . email` で3階まで一撃だお。
第5幕の『合成できるものは圏を作る』の匂いがプンプンするお

**やらない夫**：
実際 Optics は圏をなす。合成は結合的で恒等レンズ（`id`）もある。
仲間も紹介する。ここで第7-1の積・余積が効く

```haskell
-- Lens     : 積（struct）の1フィールドへのフォーカス。必ず在る
_1     :: Lens (a, b) a
-- Prism    : 余積（Either）の1枝へのフォーカス。在るかもしれない
_Left  :: Prism (Either a b) a
-- Traversal: 0個以上へのフォーカス。まとめて狙撃
each   :: Traversal [a] a

over each (*10) [1,2,3]        -- [10,20,30]（全要素を一括更新）
preview _Left (Left 5)         -- Just 5
preview _Left (Right "x")      -- Nothing
```

**やる夫**：
積へのフォーカスが Lens、余積へのフォーカスが Prism、複数が Traversal。
第7-1でやった **積と余積の対比が、そのままフォーカスの種類の対比** になってるお

**やらない夫**：
そして種明かしだ。レンズの正体は、驚くほど地味な型をしている。
**van Laarhoven 表現**──関手ひとつを受け取る高階関数だ[^18]

```haskell
type Lens s a = forall f. Functor f => (a -> f a) -> (s -> f s)
-- f に Const を入れれば view（取り出し）に、Identity を入れれば set（更新）になる
```

**やる夫**：
関手 `f` の選び方ひとつで、getter にも setter にも化けるのかお。
……つまりレンズは、**第5幕の関手だけで組み上がってる** お。
あの怪物みたいな名前の道具に、新しい原理はゼロで、
中身は関手とアプリカティブの豪華な再利用だったんだお

**やらない夫**：
それがこのツアーの主題そのものだ。次も同じことが起きる

### 8-6　畳み込みの究極形 ── Recursion Schemes

---

**やらない夫**：
第2幕の fold を極限まで一般化した名所だ。**Recursion schemes**。
動機は『再帰を毎回手で書きたくない』。具体例で刺す。式の評価器を書け

```haskell
data Expr = Lit Int | Add Expr Expr | Mul Expr Expr

eval :: Expr -> Int
eval (Lit n)   = n
eval (Add a b) = eval a + eval b      -- 再帰を手で書いている
eval (Mul a b) = eval a * eval b      -- 再帰を手で書いている
```

**やる夫**：
木を畳むのに、自分で `eval a` の再帰呼び出しを書いてるお。
pretty-print も、最適化も、木を歩く再帰を **毎回** 書き直すことになるお

**やらない夫**：
そこで『再帰している部分』を型パラメータ `r` に追い出す。
`Expr` 自身への参照を `r` に置き換えた **土台関手** を作る

```haskell
data ExprF r = LitF Int | AddF r r | MulF r r
  deriving Functor                       -- r の位置が「子ノード」

newtype Fix f = Fix (f (Fix f))          -- 再帰を1箇所に封じ込める
type Expr' = Fix ExprF
```

再帰が `Fix` の1箇所に閉じ込められた。すると畳み込みが
**catamorphism**（`cata`）という **たった1つの関数** で書ける

```haskell
cata :: Functor f => (f a -> a) -> Fix f -> a
cata alg (Fix x) = alg (fmap (cata alg) x)

-- 「1段だけの計算」= 代数 を渡すだけ。再帰は cata が面倒を見る
evalAlg :: ExprF Int -> Int
evalAlg (LitF n)   = n
evalAlg (AddF a b) = a + b               -- a, b は既に評価済みの子！再帰が無い
evalAlg (MulF a b) = a * b

eval' = cata evalAlg
```

**やる夫**：
`evalAlg` から **再帰呼び出しが消えた** お。子はもう計算済みで渡ってくるお。
しかも同じ `cata` に別の代数を渡せば……

```haskell
showAlg :: ExprF String -> String
showAlg (LitF n)   = show n
showAlg (AddF a b) = "(" ++ a ++ "+" ++ b ++ ")"
showAlg (MulF a b) = "(" ++ a ++ "*" ++ b ++ ")"

pretty = cata showAlg   -- 木を歩く再帰は一切書かず、pretty-print が手に入る
```

**再帰の骨格は1回書けば、あとは代数を差し替えるだけ** だお。
evalもprettyも最適化も、`cata` の使い回しで済むお

**やらない夫**：
逆向きに『種から木を育てる』のが **anamorphism**（`ana`）だ

```haskell
ana :: Functor f => (a -> f a) -> a -> Fix f
-- 例：1..n をカウントダウンの木に展開する、など
```

**やる夫**：
`cata` の型に `Functor f =>` が付いてるお。
**構造関手の上で畳んでる**。第5幕の関手が、ここでも土台だお。
`fold`（第2幕）＋関手（第5幕）＝あらゆる再帰の畳み込み。名前は厳ついけど、材料は全部既知だお

**やらない夫**：
出典は Meijer らの通称 *Bananas, Lenses, Envelopes and Barbed Wire*[^19]。
題にレンズも入っている。この界隈は名前遊びが好きだ

### 8-7　モナドの鏡像 ── コモナド

---

**やる夫**：
**コモナド**。モナドに co がついてるから、双対ってやつかお

**やらない夫**：
その予想を型にしろ。モナドの `pure :: a -> m a` と `join :: m (m a) -> m a` の
矢印を、ただ逆向きにしたら何になる

**やる夫**：
`w a -> a` と `w a -> w (w a)` だお。
前者は文脈から今の値を取り出し、後者は今いる場所ごとの文脈を増やす……はずだお

**やらない夫**：
合格だ。名前を付けて並べる

```haskell
-- Monad（第6幕）             Comonad（矢印が逆）
pure    :: a -> m a          extract   :: w a -> a            -- 文脈から値を取り出す
join    :: m (m a) -> m a    duplicate :: w a -> w (w a)      -- 文脈を入れ子に増やす
(>>=)   :: m a -> (a -> m b) extend    :: w a -> (w a -> b) -> w b
```

**やる夫**：
モナドが『値を文脈に **入れて**、入れ子を **潰す**』のに対して、
コモナドは『文脈から値を **取り出し**、文脈を入れ子に **増やす**』お。
完全に矢印が逆さまだお。……でも、これ何に使うんだお

**やらない夫**：
モナドが『**効果を伴う出力**』なら、コモナドは『**文脈に依存した入力**』だ。
典型は **Store コモナド**：『現在位置』と『位置→値の関数』のペア。
`extend` は『各位置で、周囲を見て新しい値を決める』を全体に一気に適用する。
1次元セルオートマトンで見せる

```haskell
data Store s a = Store (s -> a) s          -- (全体の定義, いま注目している位置)

extract (Store f s) = f s                  -- いまの位置の値

-- ルール：左右の隣を見て次世代の自分を決める（局所ルール）
rule :: Store Int Bool -> Bool
rule (Store f i) = f (i-1) /= f (i+1)      -- 左隣 XOR 右隣（Rule 90 の親戚）

step :: Store Int Bool -> Store Int Bool
step = extend rule                         -- 局所ルールを「全セルに一斉適用」
```

**やる夫**：
`extend rule` の1行で、**全セルが同時に周囲を見て更新** されるお。
ライフゲームで各セルが近傍を見て次を決めるやつ、
画像のぼかしで各ピクセルが近傍の平均を取るやつ、
スプレッドシートのセルが他のセルを参照するやつ──
全部『周囲を見て自分を決める』＝コモナドなんだお。腑に落ちたお

**やらない夫**：
第6幕の定義の矢印を反転しただけで、鏡の国がまるごと一つ手に入る。
双対の威力だ[^22]

### 8-8　フォーカスを持って歩く ── Zipper

---

**やらない夫**：
ツアーの最後は『すごいHaskell』の締めの章、**Zipper** だ。
不変データで『今いる場所』を保持して、その場を安く編集したい。
まず一番易しいリストで設計しろ。`[1,2,3,4]` の3にカーソルがあり、
左右どちらへも $O(1)$ で動きたい。何を持てばいい

**やる夫**：
左側 `[1,2]` を普通に持つと、すぐ左の2を取るのに末尾まで走るお。
なら左だけ逆順 `[2,1]` にして、**逆順の左・現在値・右**の三つを持つお。
3にいるなら `([2,1], 3, [4])`。左右移動は先頭を一個移すだけだお！

**やらない夫**：
その表現を型にしたものがリストのZipperだ

```haskell
-- (逆順の左側, フォーカス, 右側)。カーソルの左右移動が O(1)
data ListZipper a = LZ [a] a [a]

fromList (x:xs) = LZ [] x xs

right (LZ ls c (r:rs)) = LZ (c:ls) r rs      -- 右へ1歩：O(1)
left  (LZ (l:ls) c rs) = LZ ls l (c:rs)      -- 左へ1歩：O(1)
setFocus x (LZ ls _ rs) = LZ ls x rs          -- その場を編集：O(1)
```

**やる夫**：
`[1,2,3,4]` の 3 にフォーカスしてるなら `LZ [2,1] 3 [4]` だお。
左側を逆順で持つのがミソだお。左に1歩なら、逆順リストの先頭 `2` を
剥がしてフォーカスにするだけ。毎回リストを頭から辿らないから **O(1)** だお

**やらない夫**：
木に一般化すると、フォーカスした部分木に加えて
**パンくず（来た道＝各分岐で選ばなかった側と向き）** を持つ

```haskell
data Crumb a = LeftCrumb a (Tree a) | RightCrumb a (Tree a)
type TreeZipper a = (Tree a, [Crumb a])   -- (今の部分木, 戻る道)

goUp, goLeft, goRight, modify :: TreeZipper a -> TreeZipper a
```

**やる夫**：
テキストエディタのカーソル、undo/redo の編集履歴、
JSONの深い階層をカーソルで編集するツール──全部これだお。
……あれ？　これ **フォーカス付きの構造** だお。
8-7のコモナドの Store も『注目位置と全体』だったお。
**Zipper はコモナドの実例** でもあるんだお。最後の名所が、1個前と地下でつながってるお[^23]

**やらない夫**：
よく気づいた。それがこのツアーの締めくくりにふさわしい観察だ。
Zipper の発案は Huet の1997年の小論だ。名前のとおり、
服のジッパーのように構造を『開いて』フォーカスを差し込む

氷山は巨大だ。だが **プログラマが乗る必要のある部分** は水面のこの辺までだ。
注意書きを一つ。今日ずっと『型と関数の圏』と言ってきたが、
本物のHaskellには遅延評価と ⊥（停止しない計算）があって、
厳密にはきれいな圏にならないことが知られている[^8]。
実務家は『十分きれいな近似』として使う。数学の潔癖さと工学の実用性、これも国境線の一つだ

**やる夫**：
最後に旅を振り返らせてくれお。

並列バッチが壊れて **結合法則と単位元** を拾った。
平均や分散を並列化するのに **累算器を答えより豊かに** した。
nil地獄から **Option と FlatMap** を自作した。
箱のたとえが Reader で死んで、**法則そのもの** に向き合った。
Goで Monad が書けなくて、**高階型の壁** を知った。
圏を定義したら、**第2幕のモノイドが対象1個の圏** として戻ってきた。
EvilMap と CheatHead を殺して、**関手則と自然性** が検査装置だと知った。
Fish の資格審査で、**モナド則が圏の公理** に化けた。
そして今日の観光で、高等呪文が全部 **その5個の組み替え・双対・一般化** だと見えたお。

……振り返ると、やる夫が **引っかかって悔しかった場所が、全部そのまま概念の入口** だったお。
スイスイ進んでたら、何一つ身についてなかったお

**やらない夫**：
それがこの旅の設計だ。
圏論は『賢い奴が上から降らせる抽象』ではなく、
『現場の具体例が3回続いたときに、名前を付けに来てくれる整理屋』だ。
これからお前は6個目、7個目のモナド──Parser、非同期、確率──に出会う。
そのたびに今日の地図を開け。初見の怪物ではなく、旧知の構造として迎えられる。

改めて。**関数型チョットワカル、認定してやる**。
ただし本物の『チョットワカル』は『自分がどこからワカラナイかを正確に言える』という意味だ。
精進しろ。……で、次は何の障害対応だ？

**やる夫**：
goroutine が3万本湧いてるお

**やらない夫**：
それは圏論では救えん。行くぞ
